개념 정리

“줄인다”는 말은 서로 다른 비용 축을 가리킨다

같은 latency 감소라도 weight, context, call count, decoding step 중 무엇을 치는지에 따라 평가 지표가 달라진다.

① Weight

parameter memory · precision · GPU residency

② Training

full fine-tuning memory · optimizer state

③ Teacher

output/logit/router 접근 · KD 권한

④ Tool

schema token · action 후보 수

⑤ Context

action/observation/tool result 누적

⑥ Calls

235B 호출 비율 · fallback rate

⑦ Runtime

KV cache · prefix · decoding bottleneck

이 그림이 보여주는 핵심: 압축, 라우팅, runtime 최적화는 같은 층위가 아니다.

Speaker notes

  • 이 장은 시스템 패턴을 포함한 전체 지도다. KD는 weight와 teacher 의존도를 줄일 수 있지만 context 길이나 call orchestration 문제를 자동으로 없애지는 않는다.
  • vLLM이나 decoding optimization은 모델 압축이 아니라 남은 호출의 실행 비용을 낮추는 축이다.

Part 1 · 모델 자체

Quantization — 235B weight memory와 serving latency를 줄인다 논문 기반 방법론

GPTQ/AWQ의 문제정의는 FP16/BF16 serving의 메모리·대역폭 부담을 낮추는 것이다.

문제정의 → 핵심 방법

  • 문제: 235B FP16/BF16 weight는 memory residency와 bandwidth가 크고, long context에서는 KV cache 부담도 함께 커진다.
  • GPTQ: post-training quantization으로 weight precision을 낮춰 큰 모델의 serving footprint를 줄인다.
  • AWQ: activation-aware로 중요한 channel을 보호해 INT4 serving 손실을 낮추려 한다.

한계와 평가 분리

  • PPL 유지가 schema validity/action accuracy 유지와 같지 않다.
  • KV cache 자체는 별도 축이다. weight quantization만으로 긴 context 비용이 사라지지 않는다.
  • calibration 분포가 agent prompt·JSON/action 분포와 다르면 확인 필요.
General quality metricAgent metric왜 분리하는가
PPL, MMLU류 score, generic QAschema validity, action accuracy, task successagent에서는 맞는 문장보다 실행 가능한 구조가 중요
avg latency, throughputp95 end-to-end latency, retry rateworkflow 실패·재시도 비용이 평균값을 가릴 수 있음

Speaker notes

  • 이 슬라이드는 논문 기반 방법론이다. Quantization은 “student를 만든다”가 아니라 같은 큰 모델을 더 작은 precision으로 serving하는 방법이다.
  • Qwen3-235B 관점에서는 weight memory와 serving latency를 직접 치지만, agent metric은 별도로 봐야 한다.
Part 1 · 모델 자체

QLoRA — 작은 base/student를 낮은 비용으로 task adaptation 논문 기반 방법론

QLoRA는 235B를 직접 10B로 만드는 방법이 아니라 full fine-tuning memory를 줄이는 방법이다.

줄이는 것

  • full fine-tuning 시 필요한 activation·optimizer·gradient 메모리
  • base weight는 4-bit로 frozen하고 작은 LoRA adapter만 학습
  • 작은 Qwen 계열 base/student의 task-specific adaptation 비용

줄이지 않는 것

  • inference call count: workflow가 여러 번 호출하면 호출 수는 그대로
  • student의 base parameter 수 자체: 학습 방법이지 pruning이 아님
  • teacher 데이터 품질 문제: task-specific dataset은 여전히 필요
적용 조건
작은 base/student
+
task-specific dataset
→
adapter 학습

이 그림이 보여주는 핵심: QLoRA는 압축 결과가 아니라 adaptation 비용을 줄이는 학습 경로다.

Speaker notes

  • 이 슬라이드는 논문 기반 방법론이다. 쉬운 말로는 “큰 weight는 얼리고 작은 adapter만 고친다”다.
  • Qwen3-235B를 teacher로 써서 만든 response나 trajectory dataset을 작은 student에 맞출 때 QLoRA가 연결될 수 있다.
Part 2 · KD

Qwen3 teacher → student: off-policy와 on-policy를 분리한다 논문 기반 방법론

Qwen3 Technical Report는 lightweight model 학습에서 teacher response 기반 단계와 student rollout 기반 단계를 구분한다.

Off-policy distillation

  • 입력: teacher가 만든 response 또는 reasoning/no-reasoning 출력
  • 학습: student SFT target으로 사용
  • 접근 권한: teacher output만 있어도 가능

On-policy distillation

  • 입력: student가 실제 생성한 trajectory/response
  • 학습: teacher가 student output을 평가하거나 logits/KL로 보정
  • 접근 권한: logits 가능 여부에 따라 강도가 달라짐
Teacher 접근가능한 KDQwen3-235B-A22B 연결
output만 가능Response KD, preference data 생성black-box KD. MoE 내부 지식은 output에 드러난 것만 사용
teacher logits 가능Logit KD, on-policy KLtoken 분포를 더 세밀하게 맞춤. API/serving 권한 확인 필요
router/expert 접근 가능MoE-aware KD가능하다고 가정하지 않음. 접근 가능할 때만 expert coverage 논의 가능

Speaker notes

  • 이 슬라이드는 논문 기반 방법론이다. off-policy는 “teacher 답안을 보고 베끼는 단계”, on-policy는 “student가 실제로 낸 답을 teacher가 다시 잡아주는 단계”로 설명하면 쉽다.
  • Qwen3-235B-A22B는 MoE teacher이므로 output-only KD와 router/expert를 보는 MoE-aware KD는 다른 수준의 문제다.
Part 2 · KD

KD 4가지 비교 — 접근 권한과 필요 데이터가 방법을 결정한다 논문 기반 방법론

KD는 모두 “teacher 의존도를 낮춘다”로 묶이지만 필요한 권한과 보장하지 못하는 것이 다르다.

핵심: KD가 줄이는 것은 주로 큰 teacher 호출 의존도다. agent workflow의 중간 action 품질은 별도 데이터와 지표가 필요하다.

Speaker notes

  • Response KD는 선생님 답안지를 모아 학생에게 SFT하는 것이다. Logit KD는 정답뿐 아니라 “다른 후보에 얼마나 확신이 있었는지”까지 맞춘다.
  • On-policy KD는 학생이 실제로 말한 답을 선생님이 교정하는 방식이고, Preference KD는 두 답 중 무엇이 더 나은지의 선호를 학습한다.
  • 이 장은 논문 기반 방법론이고, Qwen3-235B 접근 권한이 어느 줄까지 가능한지 먼저 확인해야 한다.
Part 2 · KD

일반 KD만으로 agent workflow가 보장되지는 않는다 논문 기반 방법론

agent에서는 final answer가 맞아도 중간 action sequence가 틀리면 workflow가 깨질 수 있다.

Final answer KD

  • 입력과 최종 응답만 학습
  • 중간 계획, schema decision, observation 처리 과정은 숨겨짐
  • 겉보기 답변은 유사해도 retry·invalid JSON·wrong merge가 남을 수 있음

Trajectory KD

  • reason → act → observe 흐름을 student가 학습
  • 중간 action과 observation 반응을 supervision 대상으로 노출
  • workflow 재현성과 디버깅 가능성을 높임

이 그림이 보여주는 핵심: agent KD는 결과가 아니라 실행 경로를 학습 대상으로 올린다.

Speaker notes

  • 이 슬라이드는 논문 기반 방법론이다. agent의 실패는 “답이 조금 이상함”보다 “중간 단계가 잘못되어 전체 workflow가 중단됨”으로 나타난다.
  • Qwen3-235B teacher가 만든 trajectory가 있다면 final answer만 버리지 말고 중간 structure를 어떻게 보존할지 검토할 수 있다.
Part 2 · Agent KD

Structured Agent Distillation — [REASON]과 [ACT]를 분리한다 논문 기반 방법론

긴 reasoning token이 짧지만 중요한 action token을 압도하는 imitation loss 문제를 줄이려는 접근이다.

문제정의

  • 일반 imitation loss는 전체 token을 같은 방식으로 평균낸다.
  • reasoning span은 길고, action span은 짧지만 실행 성공에 직접적이다.
  • 결과적으로 중요한 action token supervision이 희석될 수 있다.

핵심 방법과 개선 포인트

  • trajectory를 [REASON]과 [ACT] segment로 표시
  • segment-specific loss로 reasoning과 action을 다르게 학습
  • agent가 “생각”뿐 아니라 “무엇을 호출/선택할지”를 재현하게 함

적용 조건/한계: 원 논문은 tool-call ACT 중심이다. tool-free workflow에서는 ACT를 API 호출이 아니라 schema decision, merge decision, routing decision으로 재해석해야 한다.

Speaker notes

  • 이 장은 논문 기반 방법론이다. 쉬운 말로는 “말로 생각하는 부분과 실제 버튼을 누르는 부분을 따로 채점한다”다.
  • Qwen3-235B가 긴 reasoning을 만들 수 있어도 serving output은 짧은 action으로 제한할 수 있다는 뒤의 output-length 슬라이드와 연결된다.

Part 3 · Context

Context compression — 모델을 줄여도 context가 길면 비용은 남는다 논문 기반 방법론

AgentDiet/ACON의 문제정의는 multi-step agent에서 action/observation/tool result가 누적되어 input token이 커지는 문제다.

기존 접근의 한계

  • 작은 student로 바꿔도 매 step 전체 history를 다시 넣으면 input cost가 남는다.
  • long context는 latency뿐 아니라 attention memory와 recall error를 키울 수 있다.
  • tool result가 많을수록 마지막 step의 입력이 가장 비싸진다.

핵심 방법

  • useless context: task에 기여하지 않는 관찰 제거
  • redundant context: 반복 정보 병합/요약
  • expired context: 현재 decision에 더 이상 필요 없는 과거 state 삭제

Speaker notes

  • 이 장은 논문 기반 방법론이다. AgentDiet와 ACON은 모델 자체를 줄이는 논문이 아니라 agent trajectory가 길어질 때의 context 비용을 다룬다.
  • Qwen3-235B를 student로 대체해도 누적 context가 길면 end-to-end 비용이 남는다는 점이 핵심이다.

Part 4 · Generation

Output length — decoding optimization 이전의 가장 직접적인 latency 변수 시스템 패턴

reasoning은 학습 데이터에는 포함될 수 있지만 serving output은 짧은 JSON/action으로 제한할 수 있다.

문제정의

  • 출력 token 수가 길면 어떤 decoding 최적화보다 먼저 latency가 증가한다.
  • agent serving에서는 사용자가 보는 답이 아니라 다음 step이 소비할 JSON/action만 필요할 수 있다.
  • teacher reasoning을 학습에는 쓰되 runtime 출력은 action 중심으로 제한 가능하다.

생성 안정성 지표

EOS_ratemissing_EOS_ratecap_hit_rateoutput length distributionrepetition ratep95 generation latency

주의: EOS_rate는 generation 종료와 output length 안정성 지표로만 둔다. MoE/quantized LLM에서는 router/logit/output distribution 변화 가능성이 있으므로 EOS_rate와 length distribution을 별도 평가 지표로 보는 것이 안전하다는 간접 연결만 사용한다.

Speaker notes

  • 이 장은 시스템 패턴이다. 먼저 출력 길이를 줄이고 안정화하지 않으면 decoding trick의 효과를 과대평가할 수 있다.
  • Qwen3의 thinking/no-thinking 계열과 연결할 때도 “무조건 reasoning 제거”가 아니라 serving output을 짧게 제한할 수 있다는 정도로 말한다.

Part 2 · MoE KD

  • Qwen3-235B-A22B에 바로 expert-level KD를 하자는 뜻이 아니라, 접근 권한별로 가능한 수준이 다르다는 구분이다.
Part 2 · KD 권한

Teacher 접근 권한별로 가능한 방법이 달라진다

방법을 먼저 고르는 것이 아니라, 접근 가능한 signal을 먼저 확인해야 한다.

Output only

  • Response KD
  • Preference pair 생성
  • trajectory text imitation
  • 한계: confidence·expert coverage는 보이지 않음

Logits available

  • Token/logit KD
  • On-policy KL
  • EOS/length distribution 관찰
  • 한계: MoE 내부 routing은 여전히 불투명

Router/expert available

  • MoE-aware KD
  • expert coverage/balance 학습
  • non-activated expert 활용
  • 조건: 내부 접근 가능할 때만

Qwen3-235B 연결: MoE teacher라는 사실은 중요하지만, 내부 expert 접근을 전제로 삼지 않는다. 접근 권한이 낮으면 방법도 output/logit 중심으로 제한된다.

Speaker notes

  • 이 장은 논문 기반 방법론을 권한 관점으로 재정렬한 것이다. output만 있으면 선생님 답안만 보고 배우는 셈이다.
  • router/expert 접근 가능성은 확인 필요이며, 가능하다고 말하지 않는다.

Part 6 · Domain Bridge

Domain Bridge ② — Workflow decomposition 사례 도메인 참고 사례

Decomposed Prompting, PromptChainer, R³ Prompting은 workflow를 나누는 사고법을 제공하지만 그대로 설계로 확정하지 않는다.

Decomposed Prompting

관련성: 복잡한 task를 sub-task로 나누면 작은 모델/큰 모델 분담 논의가 가능하다.

주의: decomposition 자체가 cost 절감을 보장하지 않는다.

PromptChainer

관련성: chain의 각 step input/output contract를 명시하는 관점이 agent metric과 닿는다.

주의: step이 늘면 call count와 context 누적이 증가할 수 있다.

R³ Prompting

관련성: retrieve/reason/revise류 반복 구조는 fallback·verification과 연결된다.

주의: 반복이 늘면 latency와 error propagation도 함께 늘 수 있다.

핵심: Domain Bridge는 “유사한 문제를 어떻게 분해했는가”를 보는 참고 자료이며, Profile Agent의 확정 workflow가 아니다.

Speaker notes

  • 이 장은 도메인 참고 사례다. workflow decomposition은 agent distillation과 cascade를 연결하는 개념적 다리다.
  • Qwen3-235B 관점에서는 어떤 step을 student에게 맡길 수 있는지 생각하게 해주지만, 우선순위나 정책을 제안하지 않는다.

경량화 후 확인해야 할 지표: quality와 cost를 분리해서 본다

  • EOS 관련 지표는 generation stability로만 다루며, MoE quantization 논문의 직접 주장처럼 말하지 않는다.

Part 7 · 적용 가능성

Profile Agent에 적용 가능해 보이는 지점 — 확정 설계가 아니라 연결점

여기서는 “가능성”만 요약한다. 구체적인 우선순위나 fallback 정책은 제안하지 않는다.

연결 가능성

  • 반복 schema·instruction이 있다면 output length 제한과 schema validity 지표가 중요
  • teacher trajectory가 있다면 final answer KD보다 decision/merge/action trace 보존 여부가 쟁점
  • 반복 prefix가 크면 SGLang류 prefix reuse는 runtime 가능성으로 분류
  • 긴 profile context가 누적되면 AgentDiet/ACON류 context triage가 연결됨

확정하지 않는 것

  • 어떤 step에 어떤 모델을 둘지
  • 어떤 KD recipe를 먼저 실험할지
  • fallback threshold나 accept policy
  • Domain Bridge 사례를 Profile Agent 설계로 직접 외삽하는 것

Profile Agent가 tool-free workflow라면 tool-call ACT는 schema decision / merge decision / routing decision으로 재해석해야 한다.

Speaker notes

  • 이 장은 시스템 패턴과 적용 가능성 요약이다. Profile Agent를 설계하는 장이 아니라, 앞의 taxonomy가 어디서 닿는지 짧게 연결한다.
  • Qwen3-235B 의존도는 model, context, call ratio, runtime 중 어느 축을 줄이는지에 따라 다른 계획이 필요하다.
Part 7 · 정리

Takeaways

기억해야 할 것은 특정 방법명이 아니라 분류와 평가 기준이다.

핵심 요약
  • Quantization은 235B weight memory와 serving latency를 치지만 agent metric은 별도다.
  • QLoRA는 작은 base/student adaptation 비용을 줄이지 inference call count를 줄이지 않는다.
  • Qwen3-235B-A22B는 MoE teacher이므로 output-only KD와 MoE-aware KD를 구분해야 한다.
  • Agent KD는 final answer보다 reason→act→observe 흐름을 학습 대상으로 올린다.
운영 관점
  • Tool schema token overhead와 action-space reduction은 실패 모드가 다르다.
  • 모델을 줄여도 context가 길면 비용은 남는다.
  • Serving/runtime과 decoding은 모델 압축이 아니라 남은 호출의 실행 최적화다.
  • Cascade는 큰 모델 제거가 아니라 호출 비율 감소 패턴이며 wrong accept를 함께 봐야 한다.

Speaker notes

  • 이 장은 최종 정리다. 발표를 한 문장으로 줄이면 “방법을 고르기 전에 어떤 비용 축과 어떤 teacher 접근 권한이 있는지 확인하자”다.
  • 논문 기반 방법론, 도메인 참고 사례, 시스템 패턴, serving/runtime, decoding optimization을 섞지 않는 것이 가장 중요한 방어선이다.
