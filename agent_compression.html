§0

Executive Summary

Agent LLM 경량화는 "큰 모델을 작은 모델로 바꾸는 문제"가 아니다. Agent의 비용은 모델 파라미터에만 있지 않고, tool schema · action space · trajectory context · LLM 호출 횟수 · fallback cost에 분산되어 있다. 일반 LLM 벤치마크 (PPL, MMLU, QA accuracy)에서 동등하더라도, agent workflow에서는 schema 파싱 실패·잘못된 tool 선택·multi-step 누적 오류로 인해 실패할 수 있다. 따라서 경량화의 첫 단계는 "어느 비용을 줄이려는가"를 명확히 하는 것이다.

Model

Quantization, KD, PEFT로 model serving cost (memory · latency)를 줄인다. 4-bit·LoRA 같은 표준 도구.

Tool

Tool schema와 action space를 줄여 prompt overhead와 tool-call 오류를 낮춘다. tool selection · internalization · module reuse.

Context

Trajectory history와 memory context를 압축해 multi-step 누적 비용을 줄인다. useless·redundant·expired 정보 제거.

Workflow

Cascade, router, fallback, batch decision으로 LLM call count와 큰 모델 호출 비율을 줄인다. 모델 압축과 독립적인 축.

이 문서가 답하는 4가지 질문

  1. Agent LLM 경량화가 일반 LLM 경량화와 어떻게 다른가 (§1).
  2. "줄이는 대상"을 기준으로 메서드를 어떻게 분류하는가 (§2 Taxonomy).
  3. 각 카테고리의 대표 논문, 장점, 한계, 평가 지표는 무엇인가 (§2 paper card + §3 비교 표).
  4. 상황별로 어느 메서드를 먼저 검토해야 하는가 (§5 선택 가이드).

1.2 ACBench 4-capability frame

ACBench (arXiv:2505.19433)는 압축된 LLM의 agent 능력을 측정하기 위해 4가지 capability를 정의했다. 본 문서는 이 frame을 일반 agent 평가의 기준으로 차용한다.

Capability 측정 내용 본 taxonomy에서 관련 sub-section
Action ExecutionTool 선택과 호출, argument 생성§2.4 Tool & action-space
Workflow GenerationMulti-step plan 작성과 진행§2.3 Agent trajectory KD, §2.6 system-level
Long-Context Understanding누적 trajectory와 외부 자료 처리§2.5 Context & trajectory compression
Real-world ApplicationEnd-to-end 실제 사용 시나리오 정확도§2.6 system-level + agent-level benchmark

경고: Reasoning KD는 Agent KD와 동일하지 않다

ACBench는 reasoning teacher (DeepSeek-R1 등)를 distill한 모델이 일반 reasoning은 개선되지만 tool-use accuracy가 70% → 43.6%로 떨어지는 사례를 보고했다 (ACBench Table 8, T-Eval Overall, DS-Qwen-7B vs Qwen2.5-7B FP16). 일반 instruction following이나 수학 추론이 보존돼도 tool 선택·argument 생성·schema adherence는 별도 검증이 필요하다.

외부 tool call이 없는 agent task (예: structured extraction + schema decision만 하는 agent)에는 위 수치가 직접적인 risk가 아니다. 그러나 같은 ACBench의 Workflow Generation·Long-Context Understanding 지표는 그런 task에도 직접 관련된다.

Method Category 2.1 §2.1

Model-level compression

무엇을 줄이는가: 모델 파라미터 수 · weight precision · serving memory · serving latency

핵심 아이디어: 모델 가중치를 4~8 bit로 양자화하거나, adapter (LoRA)만 학습해 학습 가능 파라미터 수를 줄인다. 추론·서빙의 단위 비용을 직접 줄인다.

적합한 상황: GPU memory 부족, serving latency가 핵심 문제, model storage 비용 절감.

Pruning 영역 (참고)

SparseGPT, Wanda 등 sparsification 기반 메서드는 ACBench가 quantization과 함께 평가한 영역이다. 본 taxonomy에서는 별도 paper card를 두지 않았으며, 일반적으로 quantization 대비 agent task에서의 quality 보존이 더 까다롭다고 보고된다 (ACBench 실험 결과 참조). 확인 필요 별도 paper-deep-reader 보강 권장.

Agent 관점 caveat

Model-level compression의 효과는 분명하지만, tool-use·schema·multi-turn robustness가 자동으로 보존되지는 않는다. ACBench 등 agent-level 평가에서 별도로 측정해야 한다.

Method Category 2.2 §2.2

General KD and Preference KD

무엇을 줄이는가: student 모델의 파라미터 수 (teacher 행동을 작은 모델에 전이). model serving cost를 줄이되 동시에 학습 신호의 질을 다룬다.

핵심 아이디어: teacher 출력을 SFT target으로 쓰거나, KL divergence를 reverse로 두거나, student가 자기 출력에서 학습 (on-policy)하거나, chosen/rejected pair로 preference를 학습한다.

적합한 상황: Teacher 모델은 있고 작은 student를 만들고 싶을 때. teacher logits 접근 여부에 따라 white-box / black-box로 갈린다.

주의 — Reasoning KD가 Agent KD를 보장하지 않는다

본 §2.2의 방법론은 모두 일반 KD / preference 학습이 주 검증 영역이다. 수학 추론이나 일반 instruction following이 보존되어도, agent의 tool 선택 / argument 생성 / schema adherence / multi-turn robustness는 별도 평가가 필요하다 (§1.2의 ACBench 경고 참조).

Method Category 2.3 §2.3

Agent trajectory distillation

무엇을 줄이는가: student의 파라미터 수 + agent reasoning/action behavior의 표현 격차. final answer만이 아니라 reason / action / observation / tool result trajectory 자체를 전이.

핵심 아이디어: teacher agent가 실제로 어떻게 step을 밟아가는지를 student에 옮긴다. final answer 매칭만으로는 작은 모델의 workflow 안정성이 보장되지 않기 때문이다.

적합한 상황: Multi-step planning이 필요한 agent. tool-use가 있거나, 누적 reasoning이 필요한 task.

Method Category 2.6 §2.6

무엇을 줄이는가: KV cache memory waste · 반복 prefix 처리 · context loading delay · structured generation execution overhead

Method Category 2.7 §2.7

무엇을 줄이는가: autoregressive decoding step 수 · token generation latency · sequential decoding dependency.

핵심 아이디어: KD와 quantization이 모델 자체를 줄이는 방법이라면, decoding optimization은 같은 모델의 autoregressive generation을 빠르게 하는 방법이다. output distribution을 유지하면서 한 forward로 여러 token을 만들거나 검증한다.

vLLM 지원 매핑 (참고)

vLLM 공식 문서 reference에 따르면 vLLM은 EAGLE / MTP / draft model / ngram / Medusa / PARD / MLP를 speculative decoding 메서드로 지원한다. 단 deploy 시점 vLLM 버전에 따라 지원 메서드와 옵션이 변할 수 있으므로 실제 사용 시 해당 버전 문서를 직접 확인해야 한다. 버전별 확인 필요

Output length와 Qwen3 Thinking Mode (decoding optimization보다 먼저 확인할 것)

가장 단순한 latency 변수는 output length다

Decoding optimization을 도입하기 전에, 먼저 확인해야 할 가장 단순한 latency 변수는 output length이다. 어떤 speculative method든 출력 token 수에 비례하는 latency를 완전히 없앨 수는 없다.

  • Agent workflow에서 어떤 step (예: Step 1 또는 Step 4)이 장황한 reasoning을 출력하면 decode latency가 커진다.
  • KD 학습 데이터에는 reasoning을 supervision으로 포함하더라도, serving output은 짧은 JSON / action으로 제한하는 것이 latency 관점에서 유리하다.
  • Qwen3는 /think와 /no_think soft switch (또는 Qwen3-Instruct-2507 / Qwen3-Thinking-2507 variant 선택)로 thinking mode와 non-thinking mode를 전환할 수 있다 (Qwen3 공식 문서 / blog 기준).
  • Hard case가 아니면 non-thinking mode 또는 제한된 reasoning budget을 검토할 수 있다.
  • 다만 reasoning을 줄이면 action quality가 떨어질 수 있으므로 schema validity, action accuracy, fallback rate를 함께 봐야 한다.

주의 — "무조건 non-thinking"이 아니다

Qwen3에서 무조건 non-thinking mode를 써야 한다고 단정하지 않는다. task difficulty와 speed requirement에 따라 thinking / non-thinking mode를 나눌 수 있다. 어려운 case는 thinking mode로, 쉬운 case는 non-thinking mode로 라우팅하는 cascade (§2.8) 패턴과 자연 연결.

Method Category 2.8 §2.8

System-level orchestration

무엇을 줄이는가: LLM 호출 횟수 · 큰 모델 호출 비율 · fallback cost · workflow step 수.

핵심 아이디어: 모델을 줄이는 것뿐 아니라 "언제 어떤 모델을 부를 것인가"가 경량화의 큰 축이다. cascade (쉬운 요청 → SLM, 어려운 요청 → LLM), router, batch decision, step merging, fallback이 여기에 속한다.

4.2 일반 벤치마크와 agent 벤치마크의 분리

일반 벤치마크 ≠ Agent 벤치마크

MMLU에서 동등한 두 모델이 schema validity rate에서 40%p 차이날 수 있다. Compression 결과를 채택할 때는 최소한 다음 4개를 함께 측정한다: ① schema validity, ② tool-call accuracy (해당하면), ③ task success rate, ④ p95 latency. cost per successful task는 단일 latency 지표보다 우선한다.

Rationale Distillation in Recommender — RDRec, SLIM

이 절은 Step별 가설이 아니라 teacher가 생성한 자연어 rationale / explanation을 supervision으로 쓰는 추천 도메인 사례를 정리한다.

RDRec: Rationale Distillation Recommender
ACL 2024 Short 다른 도메인 실험 GitHub
Xinfeng Wang, Jin Cui, Yoshimi Suzuki, Fumiyo Fukumoto · arXiv:2405.10587
핵심 아이디어
큰 LM이 user-item 관계에 대한 rationale(자연어 설명)을 생성하고, 작은 compact recommender가 그 rationale을 학습. user preference와 item attribute를 review에서 뽑아 profile에 specifies.
줄이는 대상
serving 시 큰 LM 호출 비용. rationale은 학습 supervision으로만 사용.
왜 이 조사와 관련
teacher의 자연어 reasoning을 student supervision으로 쓰는 패턴이 추천 도메인에서 적용된 사례. §2.2 KD / §2.3 trajectory KD와 같은 사고방식을 추천 task로 옮긴 예.
주의점
RDRec는 추천 ranking 정확도가 목표 — 우리 task의 평가 metric(schema validity 등)과 다름. 성능 수치 외삽 불가.

arXiv preprint 다른 도메인 실험
Yuling Wang, Changxin Tian, Binbin Hu, Yanhua Yu, Ziqi Liu, Zhiqiang Zhang, Jun Zhou, Liang Pang, Xiao Wang · arXiv:2403.04260
핵심 아이디어

줄이는 대상
serving 시 거대 teacher (예: ChatGPT-175B) 호출 비용.
왜 이 조사와 관련
"teacher reasoning을 학습으로 흡수, serving에는 작은 모델만" 패턴의 추천 도메인 변형. RDRec와 같은 가족이며 sequential recommendation에 적용.
주의점
운영상 핵심: production serving output에는 rationale을 그대로 포함하지 않고, 학습 단계 supervision으로만 쓰는 것이 안전하다. rationale 노출은 schema validity / response time / 일관성에 영향.

이 절의 일반적 메시지

teacher rationale을 supervision으로 쓰는 패턴은 추천 도메인 외에도 §2.3 (trajectory KD) / Structured Agent Distillation의 [REASON]/[ACT] 분리와 동형이다. 추천 도메인에서는 이 패턴이 ID-based / ID-agnostic 양쪽으로 확장된다는 점이 특이.

§6.3

Step 통합 vs 분리: Workflow Decomposition 근거

이 절은 "Step을 합쳐야 한다" 또는 "Step을 나눠야 한다"는 결론이 아니다. 복잡하거나 noisy한 context에서 task decomposition이 왜 유리할 수 있는지, 그리고 반대로 chain이 길어질 때의 trade-off가 무엇인지 정리한다.

Decomposed Prompting: A Modular Approach for Solving Complex Tasks
ICLR 2023 직접 실험 근거 GitHub
Tushar Khot, Harsh Trivedi, Matthew Finlayson, Yao Fu, Kyle Richardson, Peter Clark, Ashish Sabharwal · arXiv:2210.02406
핵심 아이디어
복잡한 task를 simpler sub-task로 분해, 각 sub-task를 별도 prompt · 별도 모델 · symbolic function 중 적합한 처리기로 routing. 재귀적 분해와 prompt 교체 가능.
줄이는 대상
단일 prompt의 복잡도 부담. 각 sub-task에 최적화된 처리기를 따로 둔다.
왜 이 조사와 관련
"task complexity가 커질수록 단일 prompt가 한계에 부딪힌다"는 일반 메시지가 multi-step agent workflow의 통합 vs 분리 판단의 출발점.
주의점
분리하면 모듈 수가 늘고 각 모듈 prompt 관리·테스트 비용이 증가한다. 분해가 항상 정답은 아니다.
PromptChainer: Chaining Large Language Model Prompts through Visual Programming
CHI 2022 Extended Abstracts 직접 실험 근거
arXiv:2203.06566
핵심 아이디어
prompt를 chain으로 연결하면 단일 prompt보다 투명하고 제어 가능. 동시에 intermediate output 변환 비용과 step별 debugging scaffolding이 필요하다는 점도 명시.
줄이는 대상
debugging / 점진적 개선 비용 — chain 형태가 step별 격리 디버깅을 가능하게 함.
왜 이 조사와 관련
chain 구조의 운영상 이점(투명성 / rollback)이 호출 횟수 증가의 비용과 어떻게 trade-off되는지 보여주는 사용자 연구 reference.
주의점
chain이 길어지면 schema 변환 누적 비용과 latency가 증가. 통합 호출의 latency 이점도 함께 평가해야 한다.
R³ Prompting: Review, Rephrase and Resolve for CoT Reasoning under Noisy Context
arXiv preprint 직접 실험 근거
Qingyuan Tian, Hanlun Zhu, Lei Wang, Yang Li, Yunshi Lan · arXiv:2310.16535
핵심 아이디어
noisy context에서 한 번에 답을 내기보다 Review → Rephrase → Resolve 세 단계로 나누는 방식이 더 robust. 관련 정보 추출 → 재구성 → 추론의 단계 분리.
줄이는 대상
noisy 입력에서 단일 prompt의 오류 누적.
왜 이 조사와 관련

§6.4

User Sequence / Profile Industry Case — PinnerSage · PinFM · LEADRE

이 절은 KD 논문 분류가 아니라 user behavior · user sequence · profile/recommendation에서 계산 비용과 representation 문제를 다루는 산업 사례를 정리한다. KD나 model compression 카테고리(§2.1~§2.3)와는 다른 축이며, 본문 §2.6 Serving 카테고리와 인접한 관점이다.

PinnerSage — Multi-Modal User Embedding Framework for Recommendations at Pinterest
KDD 2020 직접 실험 근거

핵심 아이디어

줄이는 대상

PinFM — Foundation Model for User Activity Sequences at Pinterest

Xiangyi Chen, Kousik Rajesh, Matthew Lawhon et al. · arXiv:2507.12704
핵심 아이디어
20B+ parameter transformer를 user activity sequence로 next-token prediction pretrain. DCAT (Deduplicated Cross-Attention Transformer)로 user sequence를 request당 1회 처리 후 KV 캐시, candidate item이 cross-attend.
줄이는 대상
user-side computation의 candidate별 반복. cross-attention dedup.
왜 이 조사와 관련
cross-attention에서의 user-side dedup 원칙이 §2.6 SGLang의 prefix 재사용 / Pinterest Dedup 블로그와 같은 sociotechnical 패턴. user 쪽 입력을 key/value로, candidate들을 query로 두는 구조의 일반 사례.
주의점
"600% throughput 향상"은 Pinterest 내부 측정치이며 hardware / batch / serving engine 조건이 다른 환경에 외삽 불가.

Fengxin Li, Yi Li, Yue Liu, Chao Zhou et al. (14 authors) · arXiv:2411.13789 · VLDB PDF
핵심 아이디어

줄이는 대상
하루 수십억 요청 규모에서의 LLM serving cost — 비실시간 path에 LLM을 두고 결과를 캐시.
왜 이 조사와 관련
latency-sensitive / latency-tolerant 모듈 분리 패턴은 §2.8 system-level의 cascade와 같은 사고방식. business objective (GMV)에 DPO를 적용한 industry 사례.
주의점
우리 시스템에 직접 적용 가능하다고 단정하지 않는다. workflow와 user 분포, business metric이 다르면 결론도 다르다.

Qwen3-235B Teacher 관점의 보강 — Qwen3 TR · Every Expert Matters

이 절은 Qwen3-235B를 teacher로 둘 때 참조할 수 있는 두 자료를 정리한다. Qwen3 TR은 family-internal distillation의 공식 recipe, Every Expert Matters는 MoE teacher의 KD 한계 분석.

Qwen3 Technical Report
arXiv preprint 직접 실험 근거 GitHub
The Qwen Team (Alibaba) · arXiv:2505.09388
핵심 아이디어
Qwen3 family (0.6B~235B-A22B MoE)의 학습 recipe 공개. 큰 flagship의 비싼 4단계 post-training 대신, 소형 모델에는 off-policy distillation(teacher의 /think 또는 /no_think 모드 response를 student SFT target) + on-policy distillation(student 생성에 대한 teacher logits KL)을 2단계로 적용. /think · /no_think soft switch로 thinking mode 전환.
줄이는 대상
소형 모델 학습 비용 — full RL training보다 효율적이라고 보고.
왜 이 조사와 관련
teacher와 student가 같은 family면 tokenizer / chat template / format이 호환된다는 운영상 이점. §2.2 KD 메서드와 §2.7 decoding optimization (thinking mode)을 연결하는 공식 reference.
주의점
위 결과는 Qwen3 family 내부에서 검증된 recipe. 우리 task에 동일 효과가 재현된다는 보장은 없음. 또한 thinking mode는 latency를 증가시키므로 §2.7의 output length 논의와 함께 봐야 한다.
Every Expert Matters — Effective KD for MoE LLMs
arXiv preprint 직접 실험 근거 (§2.2의 paper card도 참조)
Gyeongman Kim, Gyouk Chu, Eunho Yang · arXiv:2502.12947
핵심 아이디어
MoE teacher에서 활성화되지 않은 expert도 가치 있는 지식을 가진다는 관찰. KA (Knowledge Augmentation) — non-activated expert sampling, SAR (Student-Aware Router) — student feedback으로 router 학습.
줄이는 대상
표준 MoE KD가 activated expert만 본다는 한계.
왜 이 조사와 관련
Qwen3-235B-A22B는 token당 22B만 활성화되는 MoE이므로, 기존 KD로는 teacher 용량의 일부만 student에 전달된다는 일반 가설을 검증한 사례.
주의점
적용 조건: teacher 내부(expert · routing) 접근 가능 여부 확인 필요. Qwen3-235B의 내부 expert에 학습 시 routing을 강제하거나 sample할 수 있어야 KA/SAR가 의미. API-only 접근이라면 적용 불가능.

D. Serving / Decoding 관점에서의 추가 연결 가능성

아래 내용은 확정 설계가 아니라 §2.6 (Serving) / §2.7 (Decoding) 카테고리를 Profile Agent에 연결할 때의 적용 가능성 정리다.

  • SGLang 류 (repeated prefix / KV reuse) 연결 가능성: Step 1~4의 prompt에 반복되는 instruction · schema · persona · parent context가 충분히 많다면 SGLang의 RadixAttention과 연결 가능성이 있다. 단 실제 이득은 prefix 공유 길이 / 호출 빈도 / serving runtime 지원 여부에 따라 달라진다.
  • CacheGen 류 (KV cache compression / streaming) 검토: 긴 user profile이나 parent context를 반복 재사용한다면 검토 가능성이 있다. 단 KV cache를 저장·전송하는 serving architecture가 전제다. 단순히 prompt가 길다는 이유만으로 적용되지 않는다.
  • Speculative decoding 도입 전 우선 확인 사항: Step 1 또는 Step 4의 output JSON이 길거나 reasoning이 장황하다면, 먼저 output length 제한과 Qwen3 thinking / non-thinking mode 전환을 확인해야 한다. Decoding optimization은 그 다음 단계다.
  • MTP는 모델 지원 확인이 선행: Qwen3-235B의 native MTP 지원 여부 확인 전에 MTP를 적용안에 포함하지 않는다. native가 없으면 EAGLE 또는 draft model speculation을 대안으로 검토.
  • Medusa · EAGLE 학습 비용: head 또는 speculator 학습이 필요한 메서드는 학습 파이프라인 수정 권한이 있을 때만 검토. 그렇지 않으면 Lookahead Decoding처럼 학습이 필요 없는 메서드를 먼저 검토.

이 절의 한계

위 항목은 모두 적용 가능성 / 조건부 검토이며, 우선순위 매기지 않는다. 실제 채택은 측정 결과에 따른다. 본 부록은 본문 §2의 일반 방법론을 우리 task에 매핑하는 정리일 뿐, 채택 결정은 별도 실험 protocol 단계에서 이루어진다.
