개념 정리

"줄인다"의 의미: 서로 다른 비용 축

같은 latency 감소라도 weight, context, call count, decoding step 중 무엇을 치는지에 따라 평가 지표가 달라진다.

① Weight

parameter memory · precision · GPU residency

② Training

full fine-tuning memory · optimizer state

③ Teacher

output/logit/router 접근 · KD 권한

④ Tool

schema token · action 후보 수

⑤ Context

action/observation/tool result 누적

⑥ Calls

235B 호출 비율 · fallback rate

⑦ Runtime

KV cache · prefix · decoding bottleneck

TakeawayKD는 weight/teacher 의존도를 줄이지만 context 길이나 call orchestration 문제를 자동으로 없애지 않는다.

Speaker notes

  • vLLM/decoding optimization은 모델 압축이 아니라 남은 호출의 실행 비용을 낮추는 축이다.
  • Qwen3-235B 의존도는 ①~⑦ 중 어느 축을 줄이느냐에 따라 서로 다른 계획이 필요하다.

Part 1 · 모델 자체

Quantization — 235B weight memory·serving latency 절감 논문 기반 방법론

GPTQ/AWQ의 문제정의: FP16/BF16 serving의 메모리·대역폭 부담을 낮추는 것. PPL 유지 ≠ agent metric 유지.

memory ladder — 줄이는 것

FP16/BF16 — 235B full precision ~470GB+
INT8 — post-training quantization ~235GB
INT4 — GPTQ/AWQ ~118GB
  • GPTQ: 2nd-order Hessian 기반 PTQ, weight precision 낮춤
  • AWQ: activation-aware, 중요 channel 보호해 INT4 손실 완화

주의점

  • calibration 분포가 agent prompt·JSON 분포와 다르면 — 별도 검증 필요
  • KV cache는 별도 축. weight quantization만으로 긴 context 비용 미소멸
  • 4-bit quantization: PPL 약 1~3% 하락, but agent tool-use는 더 크게 영향 가능
검증 포인트PPL 유지 ≠ schema validity 유지 ≠ action accuracy 유지. 세 지표를 별도 검증 항목으로 둔다.

Speaker notes

  • GPTQ (Frantar et al., 2022), AWQ (Lin et al., 2023) — weight memory와 serving bandwidth 감소가 핵심 기여.
  • Quantization은 "student를 만든다"가 아니라 같은 큰 모델을 더 작은 precision으로 serving하는 방법이다.
  • Qwen3-235B-A22B INT4 serving 시 agent prompt·JSON calibration set 구성이 별도 검증 필요.
  • ACBench (2025): 4-bit quantization이 tool-use를 1~3%만 하락시키지만, real-world application은 10~15% 하락 가능. KD 계열보다 안전한 출발점.
Part 1 · 모델 자체

QLoRA — 작은 base/student를 낮은 비용으로 task adaptation 논문 기반 방법론

QLoRA는 235B를 10B로 만드는 방법이 아니다. full fine-tuning memory를 줄이는 adaptation 경로다.

줄이는 것
  • full fine-tuning 시 activation·optimizer·gradient 메모리
  • base weight 4-bit frozen → 작은 LoRA adapter만 학습
  • 작은 Qwen 계열 base/student의 task-specific adaptation 비용
줄이지 않는 것
  • inference call count: workflow가 여러 번 호출하면 호출 수는 그대로
  • student base parameter 수: 학습 방법이지 pruning 아님
  • 235B 자체: teacher 235B는 별도로 존재
필요한 것
  • teacher가 생성하거나 직접 수집한 task-specific dataset
  • adapter rank, target module 설정 (agent schema task에 맞게)
  • 서빙 시 adapter merge 또는 분리 선택

Speaker notes

  • QLoRA (Dettmers et al., 2023) — NF4 quantization + double quantization + paged optimizer.
  • "큰 weight는 얼리고 작은 adapter만 고친다." inference call count와 혼동하지 않는 것이 핵심.
  • adapter rank, target module 설정이 agent schema task에 적합한지 실험 필요.

Speaker notes

  • Response KD (Hinton et al., 2015 기반 계열) / Token/logit KD / GKD (Agarwal et al., 2024, on-policy) / Preference KD/PLaD 계열.
  • Qwen3-235B-A22B를 black-box teacher로만 쓸 수 있다면 response/preference 중심으로 제한된다.
  • PLaD: teacher output을 chosen, student output을 rejected로 두는 pseudo-preference pair — capacity gap이 클수록 이점 커짐.
  • 별도 검증 필요: schema validity, merge/action decision, missing_EOS_rate.
Part 2 · Agent KD

일반 KD ≠ agent workflow 보존 — trajectory KD의 이유 논문 기반 방법론

agent에서는 final answer가 맞아도 중간 action sequence가 틀리면 workflow가 깨질 수 있다.

Final answer KD — 한계

  • 입력과 최종 응답만 학습
  • 중간 계획, schema decision, observation 처리 과정 숨겨짐
  • 겉보기 답변 유사해도 retry·invalid JSON·wrong merge 남을 수 있음

  • 중간 action과 observation 반응을 supervision 대상으로 노출

Part 2 · Agent KD

Structured Agent Distillation — [REASON]과 [ACT]를 분리 논문 기반 방법론

긴 reasoning token이 짧지만 중요한 action token의 imitation loss를 압도하는 문제를 줄이려는 접근.

문제정의 → 핵심 방법

  • 일반 imitation loss는 전체 token을 같은 방식으로 평균
  • reasoning span 길고, action span은 짧지만 실행 성공에 직접적
  • 결과: 중요한 action token supervision 희석 가능
  • 해법: [REASON]/[ACT] segment 표시 → segment-specific KL loss
  • 12× 압축에서도 +4~5% Task Success Rate 보고 (원문 조건)

적용 조건/한계 재해석 필요

  • 원 논문은 tool-call ACT 중심
  • Profile Agent에서는 ACT를 schema decision / merge decision / routing decision으로 재해석해야 함 — 별도 검증 필요
reason/action 분리 예시 synthetic
{
  "reason": "candidate and existing note
    refer to the same preference",
  "action": {
    "type": "merge",
    "target_note_id": "n_013"
  }
}

reason(왜)과 action(무엇을 실행)을 구조화해서 action token supervision이 손실 계산에서 명시적으로 반영됨

서빙 시 [REASON] 출력을 끄더라도 학습 supervision으로는 유지하는 설계의 직접 근거. output length 절감 + 학습 정확도 유지.
적용 조건Profile Agent로의 외삽은 재해석 필요. tool-call ACT → schema/merge/routing decision 전환이 성능에 미치는 영향 별도 실험 필요.

Speaker notes

  • Structured Agent Distillation (arXiv:2505.13820, 2025) — segment-specific KL loss로 action supervision 강화.
  • "말로 생각하는 부분과 실제 버튼을 누르는 부분을 따로 채점한다"
  • 235B가 긴 reasoning을 만들 수 있어도 serving output은 짧은 action으로 제한할 수 있다는 output-length 슬라이드와 연결.

Part 6 · 평가

일반 LLM vs Agent LLM — 압축 영향이 비대칭이다

같은 압축 방법이라도 일반 LLM과 Agent LLM에서 평가 기준과 실패 패턴이 다르다.

일반 LLM 경량화 기준

  • 평가: PPL, MMLU, QA accuracy, generation quality
  • PPL이 유지되면 대부분 downstream task가 함께 유지됨
  • 약간 틀린 출력도 사용자가 맥락 이해 가능
  • input-output pair + 일반 instruction data로 충분

Agent LLM 경량화 기준

  • 평가: schema validity, action accuracy, task success, multi-turn robustness
  • PPL 1~3% 하락해도 real-world application 10~15% 하락 가능
  • 잘못된 JSON key 하나가 전체 workflow 파싱 실패
  • retrieval/reasoning/instruction following 능력 학습 속도가 달라 분리 학습 필요

KD 압축의 비대칭 주의점

  • reasoning KD ≠ agent KD: reasoning 능력 향상이 agentic 능력을 자동 보장하지 않는다
  • tool selection: 관련된 tool만 제공하면 execution time 최대 70% 감소 (Less is More) — but router recall이 핵심
  • format adherence: SLM은 schema/format 준수에 "struggle significantly" (Small Models Big Tasks 2025)
핵심agent 시스템 평가 지표는 일반 LLM과 독립적으로 측정해야 한다. ACBench 수치는 원문 조건에서만 유효 — speaker notes 참조.

Speaker notes

  • ACBench (2025): KD가 tool-use accuracy를 70% → 43.6%로 붕괴시킬 수 있음. reasoning KD ≠ agent KD.
  • GPTQ 4-bit / AWQ INT4는 tool use와 workflow generation 능력을 1~3% 손실로 유지 — 하지만 calibration 의존.
  • 이 수치들은 ACBench 원문 조건에서의 결과. 우리 workflow에 직접 외삽하지 않는다.
  • Agent-FLAN (2024): 능력별 학습 속도 차이와 negative sample 필요성.

Profile Agent vs Tool-calling Agent — 리스크 구조가 다르다 별도 검증 필요

tool-call accuracy는 Profile Agent의 직접 risk가 아니다. Profile Agent의 action은 외부 tool call이 아니라 schema 안의 decision이다.

Tool-calling Agent 리스크

  • tool selection accuracy — 필요한 tool을 선택하지 못함
  • API call failures — 외부 시스템 호출 실패
  • tool schema errors — 잘못된 parameter type/format
  • action space explosion — tool catalog 크기 비례
  • retry loop — failed call 반복

이 중 많은 부분이 Profile Agent에는 직접 적용되지 않는다

  • merge-decision correctness — update/merge/create/no_update 결정 오류
  • evidence preservation — 근거가 되는 신호 손실
  • long-context fidelity — 긴 profile context에서 recall 오류
  • p95 latency — 생성 길이·reasoning budget 제어

Step을 합칠 것인가, 나눌 것인가 — 트레이드오프 도메인 참고 사례

정답이 없는 트레이드오프. 입력 조건·데이터 노이즈·latency 요구사항에 따라 선택이 달라진다.

One big call (통합)

  • 짧은 latency — 호출 수 최소
  • noisy context — 모든 step의 오류가 함께 전파
  • schema risk — 복잡도가 커지면 일관성 손실 가능
  • InstructUIE 방식: 입력 노이즈가 적고 task 간 의존이 명확할 때 유리

Decomposed calls (분리)

  • subtask별 제어 — 실패 지점 명확
  • intermediate schema — 중간 output 검증 가능
  • latency 증가 — step 수만큼 호출 추가
  • R³ Prompting: 노이즈 많은 입력에서 Review→Rephrase→Resolve 분리가 유리

분기 기준 가설 (별도 검증 필요)

통합이 유리한 경우 (가설)
  • explicit review 있음 / implicit과 explicit 같은 방향
  • 로그 길이 짧음 / strong signal 존재
분리가 유리한 경우 (가설)
  • 클릭만 많고 저장/예약 없음
  • explicit review와 implicit click이 충돌
트레이드오프"Step 1~3을 합쳐야 한다"고 결론 짓지 않는다. 통합·분리는 정답이 아니라 조건에 따른 선택. 라우터 자체도 학습 대상이 될 수 있다.

Speaker notes

  • Decomposed Prompting (Khot et al., 2022, ICLR 2023), PromptChainer (Wu et al., CHI 2022), R³ Prompting (Tian et al., 2023).
  • InstructUIE (arXiv:2304.08085): 통합형 multi-task instruction tuning — 입력 노이즈가 적을 때 유리.
  • 분기 판단 기준 자체가 별도 검증 필요한 가설. "통합 vs 분리"를 자동 결정하는 라우터 자체가 또 다른 학습 대상이 된다.
  • step이 늘면 call count와 context 누적 증가 가능 — cost per successful task로 별도 검증 필요.

Profile Agent · Industry Pattern

Industry pattern — production에서는 큰 LLM을 어디에 두는가 시스템 패턴

반복되는 운영 패턴 5가지 — 설계 결정이 아니라 패턴 참고 자료. 우리 조건에서 별도 검증 필요.

패턴 1
Offline teacher labeling
큰 LLM은 production에 두지 않고, offline에서 대량 라벨·rationale 생성만.
→ Qwen3-235B로 Step별 학습 데이터 또는 rationale offline 생성 검토
패턴 2
Small online student
offline 결과를 작은 student에 distill해서 online serving은 student만으로.
→ online path에 작은 student 또는 rule/schema decision 가능성
패턴 3
Latency 분리
latency-sensitive / latency-tolerant 모듈을 분리해 무거운 모델은 tolerant 쪽에만.
→ 실시간 profile update vs 비동기 profile cleanup/batch review 분리 가능성
패턴 4
Request-level deduplication
candidate 수가 많아져도 user 쪽 입력은 한 번만 처리하고 캐시.
→ 같은 user context를 반복 처리하지 않고 candidate 비교 단계 재사용
패턴 5
Post-training denoising
추천 모델은 그대로 두고, LLM은 입력(profile)에서 노이즈만 제거하는 역할로 한정.
→ LLM을 profile noise/conflict/expired note 정리에 한정

각 패턴의 근거 사례

  • 패턴 1+2: Pinterest Search (CIKM 2024) — Llama-3-8B offline labeling → FNN student online
  • 패턴 3: LEADRE (VLDB 2025) — Hunyuan 1B offline precompute + real-time lightweight path
  • 패턴 4: Pinterest DCAT — user KV 1회 계산 후 candidate cross-attention 재사용
  • 패턴 5: CF Post-training Denoising (arXiv:2601.18009) — LLM이 user profile noise만 제거

적용 주의

  • 각 패턴은 원문 조건(데이터 규모·serving stack·목적 함수)에서 검증됨
  • 우리 latency 요구사항, 품질 기준, 데이터 조건이 다를 수 있음
  • "패턴 X를 쓰면 된다"가 아니라 "이 방향을 검토할 수 있다"
  • 실제 적용 전 별도 실험·검증 필요
산업 사례 주의산업 사례는 설계 결정이 아니라 운영 패턴 참고 자료. 우리 latency, 품질, 데이터 조건에서 별도 검증 필요.

Profile Agent · 최종 연결점

Profile Agent 연결점 요약 — 확정 설계 아님 별도 검증 필요

Profile Agent의 action은 외부 tool call이 아니라 schema 안의 decision. 각 연결 가능성은 별도 검증이 필요한 검토 항목이다.

연결 가능성 (검토 가능)

  • 반복 schema·instruction → output length 제한 + schema validity 지표
  • teacher trajectory 존재 시 → decision/merge/action trace 보존 여부가 쟁점
  • 반복 prefix가 크면 → SGLang류 prefix reuse는 runtime 가능성으로 분류
  • 긴 profile context 누적 → AgentDiet/ACON류 context triage 연결 가능
  • Step 4 → Structured Agent Distillation의 [ACT] 재해석 가능성

확정하지 않는 것

  • 어떤 step에 어떤 모델을 둘지
  • 어떤 KD recipe를 먼저 실험할지
  • fallback threshold나 accept policy
  • Domain Bridge 사례를 Profile Agent 설계로 직접 외삽
Profile Agent 의사결정 예시 synthetic example
{
  "candidate_note": "prefers quiet restaurants",
  "decision": "merge",
  "target_note_id": "n_013",
  "confidence": 0.81,
  "risk_flag": "evidence_check_required"
}

포인트: action이 구조화되면 trajectory KD 적용 가능성을 검토할 수 있다 — 별도 검증 필요

Profile Agent가 tool-free workflow라면 tool-call ACT는 schema decision / merge decision / routing decision으로 재해석해야 한다. 이 재해석 자체가 별도 검증 필요 항목이다.
핵심Qwen3-235B 의존도는 model, context, call ratio, runtime 중 어느 축을 줄이느냐에 따라 다른 계획이 필요. Profile Agent 적용 가능성: 검토 가능, 확정 아님, 별도 검증 필요.

Speaker notes

  • 이 슬라이드는 적용 가능성 분석이지 확정 설계가 아니다.
  • JSON 예시는 synthetic example임을 명시. 실제 Profile Agent 구조에서 이 action 타입들이 적합한지 별도 검증 필요.
  • tool-calling vs Profile Agent risk difference: tool selection accuracy는 Profile Agent의 직접 risk가 아니다.
  • schema validity, merge-decision correctness, evidence preservation이 Profile Agent의 직접 검증 항목이다.
Part 7 · 평가

경량화 평가: quality와 cost 분리 — 축마다 실패 모드가 다르다

압축 성공은 latency 하나로 판단하기 어렵다. agent correctness와 generation stability는 별도 측정 항목이다.

평가 축대표 지표놓치기 쉬운 실패 모드
Model qualityPPL, benchmark score, task success rategeneric score는 agent schema failure를 가릴 수 있음
Agent correctnessschema validity, action accuracy, trajectory consistency, merge-decision correctnessfinal answer만 맞고 중간 action sequence가 깨짐
Generation stabilityEOS_rate, missing_EOS_rate, cap_hit_rate, output length distribution, repetition rate긴 출력·종료 실패로 p95 latency 상승
System costfallback rate, cost per successful task, p95 end-to-end latency, retry rate작은 모델 wrong accept로 quality 손상

핵심: EOS/generation stability 지표는 MoE-KD 구조 지표가 아님. 각 논문은 원문 조건에서 보고한 결과이며 Profile Agent 성능으로 직접 외삽하지 않는다.

측정 원칙4개 평가 축을 독립적으로 측정. PPL과 schema validity는 같이 볼 수 없다. Profile Agent에서는 schema validity·merge-decision·evidence preservation·p95 latency를 직접 측정 지표로.

Speaker notes

  • 논문별 metric과 agent 운영 metric이 다르다는 점 강조.
  • EOS 관련 지표는 generation stability로만 다룬다.
  • Qwen3-235B 관련 지표 설정 시 이 4개 축을 독립적으로 측정해야 한다.

Part 7 · 정리

Takeaways — 방법명보다 분류와 평가 기준을 기억한다

기억해야 할 것은 특정 방법명이 아니라 어느 비용 축을 치는지, 어떤 teacher signal이 필요한지, 무엇이 별도 검증 필요인지.

모델/학습/KD
  • Quantization은 235B weight memory·serving latency를 치지만 agent metric은 별도 검증
  • QLoRA는 작은 base/student adaptation 비용을 줄이지 inference call count를 줄이지 않는다
  • Qwen3-235B-A22B는 MoE teacher — output-only KD와 MoE-aware KD를 구분
  • Agent KD는 final answer보다 reason→act→observe 흐름을 학습 대상으로 올린다
시스템/런타임/평가
  • Tool schema token overhead ≠ action-space reduction — 실패 모드가 다르다
  • 모델을 줄여도 context가 길면 비용은 남는다
  • Serving/runtime과 decoding은 모델 압축이 아니라 남은 호출의 실행 최적화
  • Cascade는 큰 모델 제거가 아니라 호출 비율 감소 — wrong accept 함께 봐야 함
Profile Agent 연결점
  • tool-call accuracy는 Profile Agent의 직접 risk가 아님 — schema validity, merge-decision, evidence preservation, p95 latency가 직접 리스크
  • industry 패턴 5가지: 설계 결정이 아니라 운영 패턴 참고 자료 — 별도 검증 필요
  • 확정 설계 아님, 검토 가능, 별도 검증 필요
한 문장 요약방법을 고르기 전에 — 어떤 비용 축인가, 어떤 teacher 접근 권한이 있는가, 무엇이 별도 검증 필요인가.

Speaker notes

  • 발표를 한 문장으로 줄이면 "방법을 고르기 전에 어떤 비용 축과 어떤 teacher 접근 권한이 있는지 확인하자"다.
  • 논문 기반 방법론, 도메인 참고 사례, 시스템 패턴, serving/runtime, decoding optimization을 섞지 않는 것이 가장 중요한 방어선이다.
  • Profile Agent 슬라이드(19~25)는 적용 가능성 분석이지 확정 설계가 아니다. tool-calling agent와의 리스크 차이를 명확히 유지한다.
