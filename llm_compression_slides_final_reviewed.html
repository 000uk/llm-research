"줄인다"의 의미: 서로 다른 비용 축

같은 latency 감소라도 weight, context, call count, decoding step 중 무엇을 치는지에 따라 평가 지표가 달라진다.

① Weight

parameter memory · precision · GPU residency

② Training

full fine-tuning memory · optimizer state

③ Teacher

output/logit/router 접근 · KD 권한

④ Tool

schema token · action 후보 수

⑤ Context

action/observation/tool result 누적

⑥ Calls

235B 호출 비율 · fallback rate

⑦ Runtime

KV cache · prefix · decoding bottleneck

Quantization — 235B weight memory·serving latency 절감 논문 기반 방법론

GPTQ/AWQ의 문제정의: FP16/BF16 serving의 메모리·대역폭 부담을 낮추는 것. PPL 유지 ≠ agent metric 유지.

문제정의 → 핵심 방법

FP16/BF16 — 235B full precision ~470GB+
INT8 — post-training quantization ~235GB
INT4 — GPTQ/AWQ ~118GB
  • GPTQ: 2nd-order Hessian 기반 PTQ, weight precision 낮춤
  • AWQ: activation-aware, 중요 channel 보호해 INT4 손실 완화

평가 지표 분리 — 왜 중요한가

일반 metricAgent metric
PPL, MMLU, generic QAschema validity, action accuracy
avg latency, throughputp95 end-to-end latency, retry rate
  • calibration 분포가 agent prompt·JSON 분포와 다르면 — 별도 검증 필요
  • KV cache는 별도 축. weight quantization만으로 긴 context 비용 미소멸

핵심 주의: PPL 유지 ≠ schema validity 유지 ≠ action accuracy 유지. 세 지표를 별도 검증 항목으로 둔다.

Speaker notes

  • 카테고리: 논문 기반 방법론
  • 근거: GPTQ (Frantar et al., 2022), AWQ (Lin et al., 2023) — weight memory와 serving bandwidth 감소가 핵심 기여.
  • 해석: Quantization은 "student를 만든다"가 아니라 같은 큰 모델을 더 작은 precision으로 serving하는 방법이다.
  • Qwen3 연결: Qwen3-235B-A22B INT4 serving 시 agent prompt·JSON calibration set 구성이 별도 검증 필요.
  • 별도 검증 필요: schema validity, action accuracy, p95 latency — PPL과 독립적으로 측정해야 한다.
Part 1 · 모델 자체

QLoRA — 작은 base/student를 낮은 비용으로 task adaptation 논문 기반 방법론

QLoRA는 235B를 10B로 만드는 방법이 아니다. full fine-tuning memory를 줄이는 adaptation 경로다.

줄이는 것 (확인 가능)

  • full fine-tuning 시 activation·optimizer·gradient 메모리
  • base weight 4-bit frozen → 작은 LoRA adapter만 학습
  • 작은 Qwen 계열 base/student의 task-specific adaptation 비용

줄이지 않는 것 (오해 주의)

  • inference call count: workflow가 여러 번 호출하면 호출 수는 그대로
  • student base parameter 수: 학습 방법이지 pruning 아님
  • 235B 자체: teacher 235B는 별도로 존재

이 그림이 보여주는 핵심: QLoRA는 압축 결과가 아니라 adaptation 비용을 줄이는 학습 경로다.

Speaker notes

  • 카테고리: 논문 기반 방법론
  • 근거: QLoRA (Dettmers et al., 2023) — NF4 quantization + double quantization + paged optimizer로 full fine-tuning 대비 메모리 65GB → ~12GB 수준으로 보고.
  • 해석: "큰 weight는 얼리고 작은 adapter만 고친다." inference call count와 혼동하지 않는 것이 핵심.
  • Qwen3 연결: Qwen3-235B를 teacher로 만든 response/trajectory dataset을 작은 Qwen student에 QLoRA로 맞출 때 연결 가능. 235B 자체 경량화가 아님.
  • 별도 검증 필요: adapter rank, target module 설정이 agent schema task에 적합한지 실험 필요.
Part 2 · KD

Qwen3 teacher → student: off-policy와 on-policy 구분 논문 기반 방법론

Qwen3 Technical Report: lightweight model 학습에서 teacher response 기반 단계와 student rollout 기반 단계를 분리 기술.

Off-policy distillation

  • 입력: teacher가 만든 response (reasoning·no-reasoning 출력)
  • 학습: student SFT target으로 사용
  • 접근 권한: teacher output만 있어도 가능
  • 한계: train/serve mismatch — student 실제 생성은 다를 수 있음

On-policy distillation

  • 입력: student가 실제 생성한 trajectory/response
  • 학습: teacher가 student output을 평가·KL 보정
  • 접근 권한: logits 가능 여부에 따라 강도 달라짐
  • 한계: teacher 평가 비용·exploration 안전성 확인 필요
Teacher 접근가능한 KDQwen3-235B-A22B 연결
output만Response KD, preference pair 생성black-box KD. MoE 내부 지식은 output에 드러난 것만
teacher logitsLogit KD, on-policy KLtoken 분포 세밀하게 전달. API/serving 권한 확인 필요
router/expertMoE-aware KD내부 접근 가능하다고 가정하지 않음

Speaker notes

  • 카테고리: 논문 기반 방법론
  • 근거: Qwen3 Technical Report (Qwen Team, 2025) — off-policy SFT 후 on-policy 단계로 lightweight model 학습 기술.
  • 해석: off-policy = "teacher 답안을 보고 베끼는 단계", on-policy = "student가 실제로 낸 답을 teacher가 다시 잡아주는 단계"
  • Qwen3 연결: Qwen3-235B-A22B는 MoE teacher이므로 output-only KD와 router/expert-aware MoE KD는 다른 수준이다. 내부 접근 가정 금지.
  • 별도 검증 필요: Technical Report가 기술한 학습 조건이 우리 도메인·데이터 분포와 일치하는지 원문 조건에서 확인 필요.

Part 2 · Agent KD

일반 KD ≠ agent workflow 보존 — trajectory KD의 이유 논문 기반 방법론

agent에서는 final answer가 맞아도 중간 action sequence가 틀리면 workflow가 깨질 수 있다.

Final answer KD — 한계

  • 입력과 최종 응답만 학습
  • 중간 계획, schema decision, observation 처리 과정 숨겨짐
  • 겉보기 답변 유사해도 retry·invalid JSON·wrong merge 남을 수 있음

Trajectory KD — 개선

  • reason → act → observe 흐름을 student가 학습
  • 중간 action과 observation 반응을 supervision 대상으로 노출
  • workflow 재현성과 디버깅 가능성 향상

Speaker notes

  • 카테고리: 논문 기반 방법론
  • 근거: AgentTraj-L (Liu et al., 2024), FireAct (Chen et al., 2023) 계열 — agent trajectory imitation의 동기.
  • 해석: agent의 실패는 "답이 조금 이상함"보다 "중간 단계가 잘못되어 전체 workflow가 중단됨"으로 나타난다.
  • Qwen3 연결: Qwen3-235B teacher가 만든 trajectory가 있다면 final answer만 버리지 말고 중간 structure 보존 방법을 검토할 수 있다.
Part 2 · Agent KD

Structured Agent Distillation — [REASON]과 [ACT]를 분리 논문 기반 방법론

긴 reasoning token이 짧지만 중요한 action token의 imitation loss를 압도하는 문제를 줄이려는 접근.

문제정의 → 핵심 방법

  • 일반 imitation loss는 전체 token을 같은 방식으로 평균
  • reasoning span 길고, action span은 짧지만 실행 성공에 직접적
  • 결과: 중요한 action token supervision 희석 가능
  • 해법: [REASON]/[ACT] segment 표시 → segment-specific loss

적용 조건/한계 재해석 필요

  • 원 논문은 tool-call ACT 중심
  • Profile Agent에서는 ACT를 schema decision / merge decision / routing decision으로 재해석해야 함 — 별도 검증 필요
reason/action 분리 예시 synthetic example
{
  "reason": "candidate and existing note
    refer to the same preference",
  "action": {
    "type": "merge",
    "target_note_id": "n_013"
  }
}

포인트: reason(왜)과 action(무엇을 실행)을 구조화해서 action token supervision이 손실 계산에서 명시적으로 반영됨

Speaker notes

  • 카테고리: 논문 기반 방법론
  • 근거: Structured Agent Distillation (Lu et al., 2024 계열) — segment-specific loss로 action supervision 강화.
  • 해석: "말로 생각하는 부분과 실제 버튼을 누르는 부분을 따로 채점한다"
  • Qwen3 연결: 235B가 긴 reasoning을 만들 수 있어도 serving output은 짧은 action으로 제한할 수 있다는 output-length 슬라이드와 연결.
  • 별도 검증 필요: Profile Agent로의 외삽은 재해석 필요. tool-call ACT → schema/merge/routing decision 전환이 성능에 미치는 영향 별도 실험 필요.

Part 3 · Context

Context compression — 모델을 줄여도 context가 길면 비용은 남는다 논문 기반 방법론

AgentDiet/ACON의 문제정의: multi-step agent에서 action/observation/tool result가 누적되어 input token이 커지는 문제.

기존 접근의 한계

  • 작은 student로 바꿔도 매 step 전체 history를 넣으면 input cost가 남는다
  • long context → latency 증가 + attention memory 증가 + recall error
  • tool result가 많을수록 마지막 step 입력이 가장 비싸진다
  • "Even if model is smaller, long context still costs money"

핵심 방법: 3-way triage

  • useless context: task에 기여하지 않는 관찰 → 삭제
  • redundant context: 반복 정보 → 병합/요약
  • expired context: 현재 decision에 불필요한 과거 state → 만료

  • 근거: AgentDiet (Lu et al., 2024 계열), ACON (Adaptive Context Optimization 계열) — agent trajectory 누적 context 비용 최적화.

  • Qwen3 연결: Profile Agent에서 profile context가 누적되는 구조라면 AgentDiet/ACON류 triage가 연결 가능. 적용 조건 별도 검증 필요.

Part 4 · Generation

Output length — latency를 먼저 흔드는 변수 시스템 패턴

출력 token 수는 decoding 최적화 이전에 이미 latency를 결정한다. EOS 관련 값은 generation stability 지표로만 사용.

문제정의 / 적용 조건

  • 출력 token 수 증가 → decoding trick 이전에 p95 latency 증가
  • agent serving에서는 다음 step이 소비할 JSON/action만 필요할 수 있음
  • teacher reasoning은 학습에 쓰되, runtime 출력은 action 중심 제한 가능성
  • MoE quantization 논문이 EOS_rate를 직접 다룬다고 말하지 않는다

생성 안정성 지표 (별도 측정)

EOS_rate missing_EOS_rate cap_hit_rate output length distribution repetition rate p95 generation latency

MoE quantization과의 연결: router/expert selection shift가 output distribution을 흔들 가능성 — 간접 점검 근거로만 사용

주의: EOS_rate는 MoE-KD 구조 지표가 아니라 generation 종료·length 안정성 확인 지표. MoE/quantization과의 연결은 output distribution 변화 가능성에 대한 간접 점검으로만 둔다.

Speaker notes

  • 카테고리: 시스템 패턴
  • 근거: Output length 제어 관련 연구는 generation 종료와 length 안정성의 직접 근거로만 사용.
  • 해석: 출력 길이가 길면 p95 generation latency가 커질 수 있으므로 decoding trick보다 먼저 length distribution을 본다.
  • Qwen3 연결: Qwen3 thinking mode 사용 시 reasoning token이 길어질 수 있음. serving output을 짧게 제한할 수 있는지 task별 schema와 quality metric으로 검증 필요.
  • 별도 검증 필요: EOS_rate, missing_EOS_rate, cap_hit_rate는 MoE-KD 핵심 구조 지표가 아니라 generation stability 확인 지표.

Part 6 · Domain Bridge

Domain Bridge ② — Workflow decomposition 사례 도메인 참고 사례

Decomposed Prompting, PromptChainer, R³ Prompting은 workflow를 나누는 사고법을 제공하지만 설계로 확정하지 않는다.

Decomposed Prompting

관련성: 복잡한 task를 sub-task로 나누면 작은/큰 모델 분담 논의 가능

주의: decomposition 자체가 cost 절감을 보장하지 않음

PromptChainer

관련성: chain의 각 step input/output contract를 명시하는 관점이 agent metric과 닿는다

주의: step이 늘면 call count와 context 누적 증가 가능

R³ Prompting

관련성: retrieve/reason/revise 반복 구조가 fallback·verification과 연결된다

주의: 반복 증가 → latency와 error propagation 함께 증가 가능

핵심: Domain Bridge는 "유사한 문제를 어떻게 분해했는가"를 보는 참고 자료. Profile Agent 확정 workflow가 아니다. step 분해가 call count를 늘릴 수 있으므로 cost per successful task로 별도 검증 필요.

Speaker notes

  • 카테고리: 도메인 참고 사례
  • 근거: Decomposed Prompting (Khot et al., 2022), PromptChainer (Wu et al., 2022), R³ Prompting (계열).
  • 관련성: workflow decomposition을 생각하는 참고 사례들. Qwen3-235B 관점에서 어떤 step을 student에게 맡길 수 있는지 힌트.
  • 직접 미적용 이유: step 분해가 call count를 늘릴 수 있으므로 cost per successful task로 별도 검증 필요.
  • 별도 검증 필요: Profile Agent step 구조와 매핑하기 전에 각 step의 success metric 정의 필요.

  • 해석: EOS 관련 지표는 generation stability로만 다룬다.
  • Qwen3 연결: Qwen3-235B 관련 지표 설정 시 이 4개 축을 독립적으로 측정해야 한다.

  • Qwen3 연결: Qwen3-235B를 완전히 없애는 방법과 남겨두고 최적화하는 방법을 계속 구분한다.
Part 7 · 적용 가능성

Profile Agent 연결점 — 확정 설계 아님 별도 검증 필요

여기서는 "적용 가능성"만 요약한다. 구체적인 우선순위나 fallback 정책은 제안하지 않는다.

연결 가능성 (검토 가능)

  • 반복 schema·instruction → output length 제한 + schema validity 지표 중요
  • teacher trajectory 존재 시 → decision/merge/action trace 보존 여부가 쟁점
  • 반복 prefix가 크면 → SGLang류 prefix reuse는 runtime 가능성으로 분류
  • 긴 profile context 누적 → AgentDiet/ACON류 context triage 연결 가능

확정하지 않는 것

  • 어떤 step에 어떤 모델을 둘지
  • 어떤 KD recipe를 먼저 실험할지
  • fallback threshold나 accept policy
  • Domain Bridge 사례를 Profile Agent 설계로 직접 외삽하는 것
Profile Agent 의사결정 예시 synthetic example
{
  "signal": "searched quiet restaurants",
  "profile_update": {
    "preference": "quiet atmosphere",
    "confidence": 0.82
  },
  "action": "update"
}

포인트: action이 구조화되면 trajectory KD 적용 가능성을 검토할 수 있다 — 별도 검증 필요

Profile Agent가 tool-free workflow라면 tool-call ACT는 schema decision / merge decision / routing decision으로 재해석해야 한다. 이 재해석 자체가 별도 검증 필요 항목이다.

Speaker notes

  • 카테고리: 우리 workflow로의 해석
  • 해석: Profile Agent를 설계하는 장이 아니라, 앞의 taxonomy가 어디서 닿는지 짧게 연결하는 슬라이드다.
  • Qwen3 연결: Qwen3-235B 의존도는 model, context, call ratio, runtime 중 어느 축을 줄이느냐에 따라 다른 계획이 필요하다.
  • 별도 검증 필요: JSON 예시는 synthetic example임을 명시. 실제 Profile Agent 구조에서 이 action 타입들이 적합한지 별도 검증 필요.
Part 7 · 정리

Takeaways — 방법명보다 분류와 평가 기준을 기억한다

기억해야 할 것은 특정 방법명이 아니라 어느 비용 축을 치는지, 어떤 teacher signal이 필요한지, 무엇이 별도 검증 필요인지.

모델/학습/KD
  • Quantization은 235B weight memory·serving latency를 치지만 agent metric은 별도 검증
  • QLoRA는 작은 base/student adaptation 비용을 줄이지 inference call count를 줄이지 않는다
  • Qwen3-235B-A22B는 MoE teacher — output-only KD와 MoE-aware KD를 구분
  • Agent KD는 final answer보다 reason→act→observe 흐름을 학습 대상으로 올린다
시스템/런타임/평가
  • Tool schema token overhead ≠ action-space reduction — 실패 모드가 다르다
  • 모델을 줄여도 context가 길면 비용은 남는다
  • Serving/runtime과 decoding은 모델 압축이 아니라 남은 호출의 실행 최적화
  • Cascade는 큰 모델 제거가 아니라 호출 비율 감소 패턴 — wrong accept 함께 봐야 함
  • Profile Agent 적용 가능성: 검토 가능, 확정 아님, 별도 검증 필요

한 문장 요약: 방법을 고르기 전에 — 어떤 비용 축인가, 어떤 teacher 접근 권한이 있는가, 무엇이 별도 검증 필요인가.

Speaker notes

  • 카테고리: 우리 workflow로의 해석
  • 해석: 발표를 한 문장으로 줄이면 "방법을 고르기 전에 어떤 비용 축과 어떤 teacher 접근 권한이 있는지 확인하자"다.
  • Qwen3 연결: 논문 기반 방법론, 도메인 참고 사례, 시스템 패턴, serving/runtime, decoding optimization을 섞지 않는 것이 가장 중요한 방어선이다.
