2. 권장 호출 구조 (가설)

호출묶을 단계역할
A. Signal-to-Candidate StudentStep 1 + Step 2 + Step 3raw signal → candidate note까지를 하나의 10B student 호출로 통합
B. Profile-Updater StudentStep 4 (batch)candidate note + top-k 기존 노트만 받아 update/merge/create/no_update를 한 번에 결정
C. 235B Fallbacklow confidence / schema error / 충돌 casestudent가 처리하지 못한 경우만 teacher로 escalate

⚠ 중요: 이 문서의 단계별 KD 추천은 모두 검증 필요 가설입니다

아래 Section A의 Step 1~4 권장 방법, KD 종류 선택, 호출 통합 가설은 모두 참조 논문에서 다른 도메인에 적용된 결과를 우리 workflow에 외삽한 가설입니다. 실제 우리 데이터·model size·serving 환경에서 동일한 결과가 나온다는 보장은 없으며, 각 가설은 experiment-protocol-writer를 통해 falsifiable한 실험 protocol로 변환된 뒤 측정으로 확인되어야 합니다.

3. Step별 KD 추천 한 줄 요약

Step 1 — 사용자 정보 압축

Response KD + Structured Extraction SFT (black-box). teacher가 JSON 정답을 만들고 student가 그 schema를 모방한다. 핵심은 LLM 경량화 전에 비-LLM aggregation으로 입력 token을 먼저 줄이는 것.

근거: PURE (Review Extractor), RDRec (rationale distillation)

Step 2 — 사용자 신호 병합

SFT + Preference KD (DPO 또는 PLaD). implicit와 explicit가 충돌하는 케이스에서는 "더 좋은 병합 결과"를 chosen/rejected pair로 학습해야 하며, capacity gap이 큰 235B→10B 압축에 preference KD가 적합하다는 가설.

근거: DPO, PLaD

Step 3 — candidate note 생성

Schema-constrained Response KD. factor / polarity / evidence / confidence schema의 안정적 생성이 목표. schema validity, factor classification, evidence preservation, confidence calibration이 우선.

근거: PURE, Qwen3 TR (off-policy + on-policy KL), Every Expert Matters (MoE KD)

Step 4 — 기존 노트와 병합

호출 구조 재설계 + Structured Agent Distillation + Preference KD. naive N번 비교 대신 candidate embedding으로 top-k 검색 → top-k만 한 번에 batch update. teacher 학습 데이터에는 짧은 reason을 포함하되 서빙에서는 action만 출력.

근거: Structured Agent Distillation, Agent Distillation

Section A — Step별 KD 추천 (⚠ 검증 필요 가설)
Discussion · Step 1

Step 1 — 사용자 정보 압축

raw implicit log + explicit text → implicit / explicit summary

⚠ 검증 필요
이 절의 권장 방법은 PURE의 Review Extractor 구조와 RDRec의 rationale distillation 패턴을 우리 task에 적용했을 때 잘 동작할 것이라는 가설이다.

현재 task의 본질

입력은 두 가지 채널로 들어온다. implicit log는 클릭, 저장, 체류, 길찾기, 예약 전환과 같은 사용자 행동의 로그이고, explicit text는 리뷰, 질문, 피드백, 설정과 같이 사용자가 명시적으로 남긴 문장이다.

권장 방법: Response KD + Structured Extraction SFT (black-box)

teacher인 Qwen3-235B에게 다음과 같은 JSON 형태의 정답을 만들게 한다. teacher의 logits나 attention weight를 전혀 보지 않는 black-box 방식이므로, 비공개 teacher API에서도 동작한다.

{
  "implicit_summary": [
    {
      "signal": "조용한 카페에 반복적으로 관심",
      "evidence_type": ["click", "save", "dwell"],
      "strength": "medium",
      "recency": "recent",
      "confidence": 0.74
    }
  ],
  "explicit_summary": [
    {
      "signal": "조용해서 좋았다는 리뷰 작성",
      "evidence_type": ["review"],
      "strength": "high",
      "confidence": 0.91
    }
  ]
}

주의점: LLM 경량화보다 먼저 "입력 token 경량화"

raw click log 1000개
  → 장소 category별 count
  → event type별 weight
  → recency bucket
  → conversion 여부
  → explicit signal과 충돌 여부
결합 axis 주의
"10B student로 충분하다"는 결론은 입력이 aggregated된 상태에서의 결론이다. raw log를 그대로 넣는 환경과 비교하면 일반화되지 않는다.
Discussion · Step 2

Step 2 — 사용자 신호 병합

implicit summary + explicit summary → 하나의 사용자 맥락 (evidence consolidation)

⚠ 검증 필요
Step 2를 SFT 단독이 아니라 preference KD까지 추가해야 한다는 결론은 "implicit와 explicit가 충돌하는 케이스가 우리 데이터에서 충분히 자주 나타난다"는 가정에 기반한다.

현재 task의 본질: 단순 요약이 아니라 evidence consolidation

충돌 케이스 예시
implicit: 고기집 클릭 많음
explicit: 리뷰에서 "고기 냄새 배는 곳은 싫다"

→ "고기집 선호"로 합치면 잘못된 결론이다.

권장 방법: SFT + Preference KD

candidate A: "사용자는 고기집을 선호한다."

candidate B: "사용자는 고기집을 자주 탐색했지만,
   냄새가 강한 고기집은 부정적으로 평가했다."

chosen   = B
rejected = A
전제 조건
preference KD가 효과를 내려면 chosen/rejected pair 생성 자체가 안정적이어야 한다.
Discussion · Step 3

Step 3 — candidate user note 생성

merged user context → ranking에 쓸 수 있는 schema의 candidate note

⚠ 검증 필요
Step 3의 schema-constrained 출력이 자유 텍스트보다 downstream ranking·retrieval에서 더 안정적이라는 결론은 가정이다.

평가 기준

  • schema validity — 필드 누락·타입 오류가 없는지
  • factor classification accuracy — factor가 일관되게 분류되는지
  • evidence preservation — signal 추적 가능한지
  • confidence calibration — confidence score가 실제 정확도와 상관관계를 갖는지
  • duplicate suppression — 중복 후보가 생성되지 않는지

    {
      "factor": "ambience",
      "polarity": "positive",
      "note": "조용하고 대화하기 좋은 카페를 선호",
      "evidence": [
        { "source": "review", "text": "조용해서 좋았다" },
        { "source": "save",   "count": 3 }
      ],
      "confidence": 0.86,
      "update_priority": "high"
    }
  ]
}
Discussion · Step 4

Step 4 — 기존 사용자 노트와의 병합

candidate note + existing note tree → update / merge / create / no_update

⚠ 검증 필요
Step 4의 가장 큰 가설은 "embedding 기반 top-k 검색이 candidate와 비교해야 할 기존 노트의 대부분을 cover한다"는 점이다.

권장 호출 구조

  1. candidate_note의 embedding을 생성한다.
  2. 기존 note tree에서 embedding 기준 top-k 관련 노트만 검색한다.
  3. top-k 노트만 student updater에 한 번에 넣는다.
  4. 한 번의 batch output으로 update / merge / create / no_update를 모두 결정한다.
{
  "updates": [
    {
      "action": "merge",
      "target_note_id": "n_013",
      "updated_note": "카페 방문 시 조용하고 대화하기 좋은 분위기를 선호",
      "confidence": 0.88,
      "evidence_used": ["review_17", "click_cluster_4"]
    }
  ],
  "new_notes": [],
  "no_update_notes": ["n_021", "n_034"],
  "fallback_required": false
}

권장 KD: Structured Agent Distillation + Preference KD

{
  "reason": "candidate와 n_013 모두 조용한 카페 분위기 선호를 말하므로,
             새 노트 create가 아니라 merge가 적절함",
  "action": "merge",
  "target_note_id": "n_013",
  "updated_note": "..."
}
fallback이 필수
Step 4 student의 output schema에 fallback_required: true 필드를 의도적으로 두고, low confidence·schema error·conflict 케이스를 235B로 escalate해야 안전하다.

User Profile · LLM Recommender

PURE: LLM-based User Profile Management for Recommender System

Seunghwan Bang, Hwanjun Song  ·  arXiv:2502.14541  ·  2025  ·  arxiv.org/abs/2502.14541 검증됨

논문 요약

PURE는 LLM이 사용자 행동/리뷰 데이터에서 사용자 선호를 추출해 profile을 누적적으로 관리하도록 하는 추천 시스템 구조이다. 세 가지 핵심 구성요소: ① Review Extractor — 리뷰에서 user preference와 item feature를 구조적으로 추출, ② Profile Updater — profile을 정제·갱신, ③ Recommender — 최신 profile로 개인화 추천. Amazon dataset에서 기존 LLM 기반 방법 대비 우수한 성능을 보이며 token limitation을 명시적으로 다룬다.

이 워크플로우에서의 의미

  • Step 1 매핑: Review Extractor 패턴이 teacher → student SFT 데이터 schema의 모델이 된다.
  • Step 3 매핑: profile에 누적되는 단위가 compact 단위여야 한다는 직접 근거.
  • Step 4 매핑: profile token이 무한히 커지지 않도록 update / merge 규칙이 필요하다는 점이 PURE의 token limitation 관리 구조와 같은 문제이다.

Rationale Distillation · Recommender

RDRec: Rationale Distillation Recommender

Xinfeng Wang, Jin Cui, Yoshimi Suzuki, Fumiyo Fukumoto  ·  arXiv:2405.10587  ·  ACL 2024 Short  ·  arxiv.org/abs/2405.10587 GitHub 검증됨

논문 요약

RDRec는 큰 LM이 user 행동·리뷰에 대해 생성한 rationale(왜 이 사용자가 이 item을 좋아할 가능성이 높은가에 대한 자연어 설명)을 작은 compact recommender가 학습하도록 distill한다. user preferences와 item attributes를 review에서 뽑아 profile에 specifies한 뒤, top-N 및 sequential recommendation 양쪽에서 SOTA를 달성한다.

이 워크플로우에서의 의미

  • teacher가 추론한 자연어 설명을 student의 supervision으로 쓰는 패턴이 추천 도메인에서 검증됨.
  • RDRec는 추천 ranking이 목표이고, 우리는 profile management가 목표이므로 성능 수치를 외삽할 수 없다.

Preference Optimization

DPO: Direct Preference Optimization — "Your Language Model is Secretly a Reward Model"

Rafael Rafailov, Archit Sharma, Eric Mitchell, Christopher D. Manning, Stefano Ermon, Chelsea Finn  ·  arXiv:2305.18290  ·  NeurIPS 2023  ·  arxiv.org/abs/2305.18290 GitHub (eric-mitchell) 검증됨

논문 요약

DPO는 reward model을 별도로 학습하지 않고, chosen / rejected response pair만으로 LM을 직접 최적화하는 방법이다. RLHF가 (1) reward model 학습 + (2) PPO 강화학습의 두 단계로 진행되는 것에 비해, DPO는 한 단계로 두 단계 RLHF와 동등한 목적을 달성할 수 있다는 것을 수식적으로 유도한다. KL divergence constraint를 이용한 reward maximization을 policy parameter에 대한 closed-form으로 표현한다.

이 워크플로우에서의 의미

  • Step 2 매핑: 충돌하는 implicit/explicit를 어떻게 병합할지의 선호 학습.
  • Step 4 매핑: merge vs create vs update 중 어느 action이 더 적절한지에 대한 선호 학습.
  • 핵심 전제: chosen/rejected pair를 일관성 있게 생성할 수 있어야 한다.

Preference-based KD

PLaD: Preference-based Large Language Model Distillation with Pseudo-Preference Pairs

Rongzhi Zhang, Jiaming Shen, Tianqi Liu, Haorui Wang, Zhen Qin, Feng Han, Jialu Liu, Simon Baumgartner, Michael Bendersky, Chao Zhang  ·  arXiv:2406.02886  ·  ACL 2024 Findings  ·  arxiv.org/abs/2406.02886 검증됨

논문 요약

PLaD는 KD에서 teacher output을 그대로 모방하는 방식(sequence-level mimicry)이 teacher와 student의 capacity gap이 클 때 학습 신호로서 한계가 있다는 점을 지적한다. 대신 teacher output을 chosen, student output을 rejected로 두는 pseudo-preference pair를 만들고, student가 "상대적으로 더 좋은 출력을 선호"하도록 ranking loss로 sequence likelihood를 재보정한다.

이 워크플로우에서의 의미

  • Step 2 / Step 4 매핑: 235B→10B capacity gap에서 PLaD의 ranking loss가 단순 SFT보다 유리할 가능성.
  • DPO와의 차이: PLaD는 teacher/student 자기 자신의 출력으로 pseudo-pair를 만든다. 둘은 결합 가능하다.

Model Family · Distillation Recipe

Qwen3 Technical Report

The Qwen Team (Alibaba)  ·  arXiv:2505.09388  ·  2025  ·  arxiv.org/abs/2505.09388 GitHub (QwenLM/Qwen3) 검증됨

논문 요약

Qwen3 Technical Report는 Qwen3 model family (0.6B ~ 235B-A22B MoE, dense + MoE 통합)의 학습 recipe를 공개한다. Qwen3-235B-A22B는 MoE 구조(128 experts / 8 activated, no shared experts, global-batch load-balancing)로, activation parameter는 22B이다.

Flagship 모델의 4단계 post-training pipeline:

  1. Long-CoT cold start: 기본 reasoning pattern 구축
  2. Reasoning RL (GRPO): 수학/코드 강화학습
  3. Thinking mode fusion: CoT data와 non-CoT data를 합치고 /think//no_think control token 도입
  4. General RL: instruction following, preferences, agent, multilingual

Small model용 Strong-to-Weak Distillation (2단계): flagship의 비싼 4단계 pipeline 대신, (1) off-policy distillation: teacher가 /think//no_think 모드로 생성한 response를 student SFT target으로 사용, (2) on-policy distillation: student 자신의 생성물에 대해 teacher logits와의 KL divergence를 최소화. on-policy distillation이 full RL training보다 accuracy와 training cost 양쪽에서 우수하다고 보고한다.

Thinking budget 기능: 최대 reasoning token budget을 설정하면 budget 도달 시 강제로 reasoning을 중단하고 accumulated reasoning으로 답을 생성한다.

이 워크플로우에서의 의미

  • teacher (Qwen3-235B-A22B MoE) → student (Qwen3 dense small) 매칭이 family-internal distillation으로서 자연스럽다. tokenizer / chat template 차이가 적어 SFT 데이터 호환성이 높다.
  • off-policy + on-policy distillation 두 단계는 우리 Step 1~3 SFT 데이터 구축에도 직접 적용 가능: 먼저 235B response를 target으로 SFT(off-policy), 이후 10B student 자신의 출력에 대해 235B logit alignment(on-policy KL)로 fine-grained alignment를 추가한다.
  • thinking budget 개념: Step 4 fallback 판단에서 "student가 얼마나 reasoning을 소비했는지"를 confidence signal로 사용하는 설계의 참고가 된다.
  • Every Expert Matters와의 결합: Qwen3-235B가 MoE이므로, standard off-policy distillation이 activated expert(22B)만 사용하는 한계를 KA/SAR로 보완할 수 있다는 가설.

MoE KD · 신규 추가

Every Expert Matters: Towards Effective Knowledge Distillation for Mixture-of-Experts Language Models

Gyeongman Kim, Gyouk Chu, Eunho Yang  ·  arXiv:2502.12947  ·  February 2025  ·  arxiv.org/abs/2502.12947 검증됨

논문 요약

기존 KD 방법들은 MoE teacher에서 token당 활성화된 expert(activated experts)만을 distillation 대상으로 삼는다. 이 논문은 활성화되지 않은 expert(non-activated experts)도 학생 모델에게 가치 있는 지식을 제공할 수 있다는 것을 핵심 발견으로 제시한다.

두 가지 핵심 방법:

  • KA (Knowledge Augmentation): 활성화된 expert 외에 비활성화된 expert들도 여러 번 sampling해서 distillation signal을 풍부하게 만든다.
  • SAR (Student-Aware Router): 모든 expert를 사용하되, router가 추론 시점의 routing이 아니라 학생 모델 학습에 가장 도움이 되는 expert weighting을 학생 feedback으로 학습한다.
우리 Qwen3-235B MoE teacher에 대한 직접 함의
Qwen3-235B-A22B는 총 235B parameters에서 per-token activation은 22B인 MoE이다. 기존 KD로는 각 token에 활성화되는 22B worth의 expert 지식만 student에 전달된다. KA와 SAR는 나머지 213B의 non-activated expert 지식도 활용하는 구조이므로, MoE teacher에서 student를 학습할 때 매우 직접적으로 적용 가능하다.

이 워크플로우에서의 의미

  • MoE teacher KD의 underexplored 문제: 235B MoE teacher → 10B dense student에서 표준 response KD는 activated expert만 사용해 teacher 용량의 ~9%만 활용. KA/SAR로 보완 가능.
  • Step 1~3 SFT 데이터 품질 향상: 같은 입력에 대해 다수 expert에서의 변형을 KA로 모아 더 풍부한 학습 신호를 만든다.
  • Step 4 action 학습: SAR처럼 "student가 잘 학습하도록 teacher representation을 조정"하는 아이디어는 단순 response copy보다 adaptive한 데이터 생성의 방향성을 제시.
적용 시 주의: 학습 비용 증가
KA는 inference 시 non-activated expert를 추가로 sample해야 하므로 teacher data 생성 비용이 증가한다. SAR는 router 학습을 위해 student feedback loop가 필요하다. 우리 use case에서는 teacher data 생성이 offline batch이므로 KA는 적용하기 쉽고, SAR는 추가 engineering이 필요하다.

Structured Agent Distillation은 agent trajectory를 단순 token-level imitation으로 학습하지 않는다. trajectory를 [REASON] span과 [ACT] span으로 나누고, 각 span에 대해 separate KL-divergence penalty를 적용해 reasoning fidelity와 action consistency를 따로 보존한다. token-level / imitation-learning baseline 대비 +4~5% Task Success Rate, 12× 압축에서도 latency 감소를 보고한다.

Section C — Industry 적용 사례
Discussion · Industry Pattern

Industry에서 반복되는 공통 패턴

"LLM teacher offline · small student online" 구조와 그 변형들

반복되는 해결 패턴

  1. Offline teacher labeling: 큰 LLM은 production에 두지 않고, offline에서 대량의 라벨이나 rationale을 생성하는 데만 사용한다.
  2. Small online student: 그 결과를 작은 student model에 distill해서 online serving은 student만으로 처리한다.
  3. Latency-sensitive / latency-tolerant 분리: 응답 시간이 중요한 모듈과 그렇지 않은 모듈을 분리해 무거운 모델은 latency-tolerant 쪽에만 둔다.
  4. Request-level deduplication: candidate 수가 많아져도 user 쪽 입력은 한 번만 처리하고 캐시한다.
  5. Post-training denoising: 추천 모델 자체는 그대로 두고, LLM은 입력(profile)에서 노이즈만 제거하는 역할로 한정한다.

Pinterest — Scaling Recommendation Systems with Request-Level Deduplication

Matt Lawhon, Filip Ryzner, Kousik Rajesh  ·  Pinterest Engineering Blog  ·  medium.com (Pinterest Engineering) 검증됨

요약

Pinterest recommendation funnel에서 같은 user data가 retrieval / ranking / candidate scoring 단계에서 반복 복사된다. 핵심 구조: context transformer가 user의 action sequence를 request 당 한 번만 처리하고, 각 layer의 key-value (KV) representation을 캐시한다. 이후 각 candidate item은 이 캐시된 user-history KV에 cross-attend하여 redundant recomputation을 제거한다 — DCAT (Deduplicated Cross-Attention Transformer).

측정 결과 (Pinterest 내부 보고):

  • Storage: user-heavy feature column에서 10~50x compression
  • Training: 4x retrieval speedup, 2.8x ranking speedup
  • Serving: 7x ranking throughput 증가 → 비례 비용 증가 없이 100x 더 큰 모델 배포 가능

⚠ Pinterest 내부 측정치이며, hardware/batch/sequence/serving engine 조건이 다른 환경에 외삽 불가.

이 워크플로우에서의 의미

  • 우리 Step 4의 "candidate vs 기존 노트 N개" 비교에서 user-side 입력을 candidate마다 다시 보내지 않고 KV-캐시로 재사용하는 설계의 근거.
  • custom Triton kernel을 통한 GPU-side 최적화 — 우리 serving stack이 Triton을 지원하면 같은 방향으로 최적화 가능.
  • KD와 독립적인 system-level 최적화이므로 모델 교체 없이 먼저 적용 가능.

PinFM: Foundation Model for User Activity Sequences at a Billion-scale Visual Discovery Platform

Xiangyi Chen, Kousik Rajesh, Matthew Lawhon et al.  ·  arXiv:2507.12704  ·  ACM RecSys 2025  ·  arxiv.org/abs/2507.12704  ·  ACM DL 검증됨

요약

Pinterest에서 500M+ 사용자 규모의 user activity sequence를 위한 foundation model이다. 20B+ parameter transformer를 next-token prediction objective로 pretrain하고, home feed / search / ads / notifications 등 여러 downstream application에 fine-tune한다. LEADRE: Multi-Faceted Knowledge Enhanced LLM Empowered Display Advertisement Recommender System

Fengxin Li, Yi Li, Yue Liu, Chao Zhou et al. (14 authors)  ·  arXiv:2411.13789  ·  VLDB 2025 Vol.18, p4763  ·  arxiv.org/abs/2411.13789  ·  VLDB PDF 검증됨

요약

Tencent의 display advertising(WeChat Channels, WeChat Moments)에 LLM을 적용한 production 사례. LLM backbone: Hunyuan 1B. 핵심 문제는 implicit intent, sparse behavior, LLM-광고 시스템 knowledge gap, 하루 수십억 요청 규모에서의 deployment cost.

세 가지 핵심 기술:

  1. Intent-Aware Prompt Engineering: user intent signal + ad content + business metadata를 통합한 (Prompt, Response) pair fine-tuning.
  2. Advertising-Specific Knowledge Alignment: auxiliary fine-tuning + DPO로 semantic relevance와 business value(GMV) 양쪽에 align.
  3. Hybrid Deployment: latency-tolerant path = LLM이 user interest profile / ad representation을 비동기로 precompute·캐시. latency-sensitive real-time path는 경량 path 사용. TensorRT-LLM acceleration.

Online A/B 결과 (Tencent 내부): WeChat Channels +1.57% GMV, WeChat Moments +1.17% GMV.

이 워크플로우에서의 의미

  • latency-sensitive / latency-tolerant 분리: 우리 Step 4(실시간 update) vs 주기적 정리(비동기 batch)의 분리 가설을 수십억 requests/day 규모 production에서 검증한 사례.
  • DPO로 business objective align: 단순 semantic relevance가 아니라 GMV라는 business metric에 DPO를 적용. 우리 Step 2/4의 preference pair chosen 기준을 "downstream ranking 성능을 더 높이는 병합"으로 잡는 설계와 같은 방향.
  • high-value user 우선 GPU 할당 → active user에 큰 teacher를 더 자주 호출하는 분기 가설로 매핑 가능.

Post-Training Denoising of User Profiles with LLMs in Collaborative Filtering Recommendation

Ervin Dervishaj, Maria Maistro, Tuukka Ruotsalo, Christina Lioma  ·  arXiv:2601.18009  ·  arxiv.org/abs/2601.18009 검증됨 — 2026-05-29 arXiv에서 직접 확인

arXiv 직접 조회 결과 실제 등재 논문임이 확인되었다. 정식 제목: "Post-Training Denoising of User Profiles with LLMs in Collaborative Filtering Recommendation". 저자: Ervin Dervishaj, Maria Maistro, Tuukka Ruotsalo, Christina Lioma.

요약

implicit feedback 데이터의 노이즈 문제를 다룬다. CF recommender의 architecture나 training procedure를 바꾸지 않고, LLM을 post-training denoising 도구로만 사용한다. LLM에 (i) user profile (interactions), (ii) candidate item, (iii) CF rank를 prompt로 주고, candidate rank가 더 좋아지는 방향으로 user profile에서 불필요한 interaction을 제거한다. 여러 dataset과 LLM에서 baseline user profile 대비 최대 13% 효과 개선을 보고한다.

이 워크플로우에서의 의미

  • LLM이 추천을 직접 수행하지 않고 profile에서 노이즈만 제거 — 우리 Step 4의 "오래된 note decay / 충돌 해결" 단계에 적용 가능.
  • 기존 시스템(CF recommender)을 건드리지 않는 minimal-risk LLM 통합 방식.
  • "candidate rank가 좋아지는 방향"이라는 측정 가능한 signal로 LLM이 무엇을 지워야 할지 학습할 수 있다는 점은, 우리 Step 4의 conflict 해결 학습에도 같은 신호 설계를 적용할 수 있음을 시사.
