개념 정리

"줄인다"는 말은 7개의 다른 비용을 의미한다

Qwen3-235B 의존도는 단일 축이 아니다 — 어느 비용을 줄이려는지 먼저 정한다.

① Model size

parameter · weight precision · serving memory

② Training cost

fine-tune time · GPU memory · 학습 데이터 비용

③ Teacher dep.

teacher logits 접근 / serving 호출 의존

④ Tool / action space

schema token · tool 후보 수

⑤ Context length

누적 trajectory · long-context cost

⑥ Call count

workflow 안에서 큰 모델 호출 횟수 · fallback rate

⑦ Serving / decoding

KV cache · prefix · autoregressive step 수

같은 "latency 감소"라도 ①을 줄여 얻는 것과 ⑦을 줄여 얻는 것은 작용 지점이 다르고, 평가 지표도 다르다.

Speaker notes

  • 이 7개 축은 발표 전체의 mental map이다 — 슬라이드마다 "어느 축을 줄이는 방법인가"를 의식하면서 들어달라.
  • 예: KD는 ①+③을 줄이지만 ⑤·⑦은 그대로 남는다. vLLM은 ⑦만 줄인다. Cascade는 ⑥만 줄인다.
  • 이 분리가 안 되면 "그래서 뭐가 가장 빠른가" 같은 무의미한 비교가 나온다.
  • 일반적인 시스템 사고: latency·cost 함수를 분리해서 보는 것.

Quantization — weight precision을 낮춘다 논문 기반

GPTQ · AWQ가 대표적 — 같은 모델 그대로 두고 메모리와 latency를 직접 줄인다.

메모리 절감 layer (개념도)
FP16 base100% memory
INT8~50%
INT4 (GPTQ / AWQ)~25%, 3–4.5× speedup*

* 원문 조건(A100 / A6000)에서 보고된 수치, hardware·batch에 따라 다름

GPTQ (ICLR 2023)

second-order info 기반 one-shot weight quantization. 175B를 A100 약 4시간에 양자화.

AWQ (MLSys 2024)

activation 크기 기준 salient 1% weight를 per-channel scaling으로 보호. backward 없음.

한계

일반 PPL·MMLU가 유지돼도 schema validity / action accuracy는 자동 보존되지 않는다. Agent 평가 별도 필요.

실험에서는 4-bit가 일반적 sweet spot. 3-bit 이하는 task별 quality drop이 큼 — 단정하지 말고 측정.

Speaker notes

  • Quantization은 가장 표준화된 카드. 학습 없이 weight precision만 낮춘다.
  • 쉬운 말: "16비트 부동소수점 → 4비트 정수로 저장. 모델 구조 안 바뀌고 메모리만 1/4."
  • 논문 기반 — GPTQ / AWQ 둘 다 직접 실험 결과 있는 venue accepted 논문.
  • Qwen3-235B 연결: 235B를 INT4로 양자화하면 메모리는 줄지만, agent task의 schema 유지 여부는 ACBench 같은 별도 평가 필요. "일반 LLM 벤치마크 = agent 벤치마크"가 아니다.

PEFT / QLoRA — 학습 비용을 줄인다 논문 기반

서빙 모델을 줄이는 도구가 아니라 "큰 모델을 작은 GPU에서 fine-tune하는" 도구.

QLoRA (NeurIPS 2023)

  • 4-bit frozen base + LoRA adapter
  • NF4 · double quantization · paged optimizer

중요한 분리: Quantization = serving 비용 절감. QLoRA = 학습 비용 절감. 둘은 "모델을 줄인다"라는 표현 아래 자주 혼동된다.

Speaker notes

  • 흔한 오해: "QLoRA로 모델이 작아진다" — 아니다. base는 그대로다. 학습 가능한 파라미터만 작다.
  • 쉬운 말: "큰 모델은 얼려두고, 옆에 작은 adapter만 학습한다. 학습 시 GPU memory가 적게 든다."
  • 논문 기반 — Dettmers et al., NeurIPS 2023.
  • Qwen3 연결: 우리가 student를 학습한다면 (예: Qwen3-4B에 SFT), QLoRA가 GPU 자원 측면에서 표준 선택지.
Part 1 · 모델 자체

Qwen3 teacher → student distillation 관점 논문 기반 도메인 reference

Qwen3 Technical Report가 family-internal distillation의 공식 recipe를 공개했다.

Qwen3 TR 공식 recipe (소형 모델)

  • Off-policy distillation — teacher의 /think 또는 /no_think response를 student SFT target
  • On-policy distillation — student 생성에 대한 teacher logits KL
  • 2단계 → 소형 모델 학습 비용이 full RL 대비 효율적 (원문 보고)

같은 family라서 얻는 이점

  • tokenizer 호환 — token-level KD가 자연스러움
  • chat template 호환 — 같은 system prompt 형식
  • /think·/no_think soft switch 동일

주의 1: Qwen3-235B-A22B는 MoE이므로 표준 KD는 activated expert(22B)만 본다. → 슬라이드 25에서 "Every Expert Matters" 한계 분석.

주의 2: Qwen3 TR 자체 결과는 Qwen3 task에서 검증된 것 — 우리 task에 같은 효과가 재현된다는 보장은 없음.

Speaker notes

  • 이 슬라이드는 "Qwen3-235B를 teacher로 두면 어떤 recipe가 있는가"에 대한 공식 reference.
  • 같은 family의 이점은 운영상 매우 큼 — tokenizer가 다르면 token-level KD 자체가 어렵다.
  • 논문 기반 — Qwen3 TR (Alibaba, arXiv:2505.09388).
  • 그러나 235B-A22B의 MoE 특성 때문에 표준 KD는 한계가 있다는 점을 슬라이드 25에서 보강.

Part 2 · KD

Token / logit KD — 분포 자체를 매칭한다 논문 기반

Forward KL (coverage) · Reverse KL (mode-seeking, MiniLLM)

Forward KL

KL(teacher || student) — teacher가 부여한 모든 모드를 student가 cover하려 함. 과도하게 평탄해질 위험.

Reverse KL — MiniLLM (ICLR 2024)

KL(student || teacher) — student가 teacher의 주요 모드에 집중. mode-seeking.

두 방향의 차이 (개념도)

Forward KL

여러 가지 답을 다 흉내내려 함 — diversity↑, 헛소리 가능성↑

Reverse KL

teacher가 가장 확신하는 답에 집중 — 일관성↑

필요 조건: teacher logits 접근. API-only는 적용 불가. self-hosted Qwen3-235B 같은 환경에서만 가능.

MiniLLM 실험: 760M student가 1.5B teacher를 UnNI Rouge-L에서 능가 (원문 조건). agent task 직접 검증은 아님.

Speaker notes

  • 핵심 분리: forward KL은 다양성을, reverse KL은 집중을 학습 — agent에서는 reverse가 보통 더 안정적.
  • 쉬운 말: "teacher가 어느 단어에 몇 % 확률을 줬는지까지 흉내낸다."
  • API-only면 못 쓴다 — logits이 필요하다는 운영 조건이 핵심.
  • Qwen3-235B를 우리 GPU에 띄울 수 있어야 가능. Hosted endpoint 의존이면 슬라이드 8 (Response KD)로 회귀.
Part 2 · KD

On-policy KD — student의 자기 생성에서 학습 논문 기반

GKD (ICLR 2024) · 외부 표시는 같지만 분포 차이(exposure bias)를 직접 다룬다.

Off-policy vs On-policy

Off-policy (Response / Token KD)

teacher가 만든 sequence로 학습. 실제 inference 때 student가 만드는 분포와 mismatch.

On-policy (GKD)

student가 자기가 생성한 sequence에 teacher가 feedback. multi-step에서 누적되는 exposure bias를 직접 다룸.

왜 agent에서 의미가 큰가

  • Agent는 자기 출력이 다음 step의 입력이 된다 → exposure bias가 누적
  • off-policy로만 학습한 모델은 첫 step은 잘하지만 4–5 step 뒤 무너지는 패턴이 흔함
  • Qwen3 TR도 on-policy distillation을 2단계로 포함

대가

student가 학습 중 실시간 sampling을 해야 함 → 학습 비용 증가, teacher logits 필요.

Speaker notes

  • 쉬운 말: "student가 직접 답을 써보고, teacher가 옆에서 빨간펜으로 고쳐준다."
  • multi-step agent에서 가장 중요한 KD 변형 중 하나 — 누적 오류를 줄이는 데 직접 작용.
  • 논문 기반 — GKD (Agarwal et al., ICLR 2024). Qwen3 TR도 같은 패턴 사용.
  • 대가: 학습 시 student forward + teacher forward 모두 필요 — GPU 비용이 표준 SFT보다 큼.

Preference KD — 좋은 / 나쁜 쌍으로 학습 논문 기반

DPO · PLaD · PAD — RLHF 없이 chosen/rejected pair로 student를 정렬한다.

DPO

RLHF의 reward model + PPO를 closed-form 1단계로. chosen/rejected pair만 필요.

NeurIPS 2023

PLaD

teacher output = chosen, student output = rejected라는 pseudo-preference pair. 외부 annotation 불요.

ACL 2024 Findings

PAD

3-stage: high-temp sampling → intrinsic preference 평가 → align.

한계: chosen/rejected 자체 일관성이 학습 신호의 상한. pair를 안정적으로 만들 수 없으면 효과 약화.

Speaker notes

  • 정답이 분명하지 않을 때 유용 — "더 좋은 쪽"만 알면 학습된다.
  • 쉬운 말: "정답 1개를 따라하는 게 아니라, 두 후보 중 좋은 쪽을 고르도록 학습."
  • 운영 위험: chosen/rejected pair를 자동 생성하면 그 일관성이 학습 신호의 상한 — preference 데이터 품질이 핵심.
  • Qwen3 연결: 충돌하는 신호(예: 명시적 vs 암묵적)를 처리하는 step이 있다면 후보가 될 수 있다.

  • 이 표가 KD 파트의 결론 — 라벨 1개를 기억하라면 "logits 접근 가능 / 불가능"이라는 줄.
  • Qwen3-235B가 self-hosted면 4가지 다 가능. API-only면 사실상 Response KD + Preference KD만.
  • 일반 LLM 벤치마크에서 KD가 잘됐다는 것이, 우리 agent task에서도 잘된다는 보장은 아니다 (다음 슬라이드).
  • 한 번 강조: KD는 "어떤 형태인지"보다 "데이터를 어떻게 만들었는지"가 효과를 결정한다.
Part 3 · Agent-specific

일반 KD만으로 부족할 수 있는 이유 논문 기반

PPL · MMLU가 동등해도 agent step에서 무너지는 failure case가 있다.

ACBench (arXiv:2505.19433)의 KD failure case

4-bit quantization은 workflow generation을 1–3% 손실로 보존했지만, reasoning-teacher (DeepSeek-R1) KD는 tool-use 정확도를 70% → 43.6%로 떨어뜨렸다 (DS-Qwen-7B vs Qwen2.5-7B FP16, T-Eval Overall, Table 8).

우리 workflow는 tool-calling agent가 아니므로 이 수치를 그대로 외삽하지 않는다. 그러나 "일반 reasoning은 보존되었어도 agent capability가 별도로 망가질 수 있다"는 메시지는 동일하게 적용된다.

Agent에서의 실패 양상

  • 잘못된 action 선택 (final answer는 맞아도 중간이 틀림)
  • JSON / schema 파싱 실패
  • multi-step에서 첫 step은 OK인데 4–5 step 후 누적 오류
  • SLM이 "format adherence struggle significantly" 보고 (Small Models Big Tasks)

그래서 필요한 것

  • 최종 답안만이 아니라 trajectory (reason → act → observe) 자체를 전이
  • reasoning과 action을 분리해서 학습 신호 부여
  • agent-level 지표 (schema validity · task success · multi-turn robustness) 직접 측정

Speaker notes

  • 이 슬라이드의 핵심: "MMLU 잘하면 agent도 잘한다"는 가정은 ACBench가 정량적으로 반박했다.
  • 우리 workflow는 tool-calling agent가 아니지만 schema validity, multi-step robustness는 직접 위험.
  • 그래서 일반 KD에서 Agent KD로 넘어간다 — 다음 슬라이드.
  • 논문 기반 (ACBench). 한 줄 인용은 원문 Table 8 조건과 함께 사용.

Structured Agent Distillation — [REASON] / [ACT] 분리 논문 기반

reasoning과 action에 서로 다른 loss를 부여한다 (arXiv:2505.13820).

Trajectory를 두 segment로 분리
[REASON]
"왜 이 행동을 하는가"
+
[ACT]
"무엇을 할 것인가"

각 segment에 별도 KL loss → reasoning fidelity와 action consistency를 각각 보존

왜 분리하는가

  • 두 segment의 정보량 / 분포가 다름
  • 일반 imitation loss는 긴 reasoning이 짧은 action을 압도
  • action 부분의 schema·argument 정확도가 자동 보존되지 않음

평가

ALFWorld · HotPotQA-ReAct · WebShop에서 token-level / imitation baseline 대비 task success rate 향상 보고 (원문 조건 명시).

주의: 원문의 [ACT] 정의는 외부 tool call 단위. tool-free workflow에서는 [ACT]를 "schema 안의 결정값"으로 재해석해야 같은 사고방식이 적용된다.

Speaker notes

  • 같은 문장 안에 있어도 "왜 그렇게 했는가"와 "무엇을 했는가"는 학습 비중을 다르게 줄 필요가 있다 — 그 발견을 명시한 논문.
  • 쉬운 말: "reasoning은 길고 다양하고, action은 짧고 정확해야 한다. 같은 loss로 학습하면 한쪽이 묻힌다."
  • 논문 기반 — arXiv:2505.13820. 단 [ACT] 정의는 tool call 한정이라는 한계 있음.
  • Qwen3 연결: 우리 workflow의 step별 출력에도 "긴 설명 + 짧은 결정값" 구조가 있다면 같은 사고방식이 의미를 가진다.

Tool schema / token overhead reduction 논문 기반

모델 크기가 아니라 매 호출마다 prompt에 들어가는 tool description의 비용.

Before

User query + 전체 tool catalog (수십~수백 schema) + 긴 description

→ prompt token의 상당 부분을 tool schema가 차지
(보고치: 입력의 82.6% 사례 — Internalizing Tool Knowledge)

After (3가지 접근)

  • Internalization — tool 지식을 모델 가중치에 학습 (QLoRA)
  • Tool selection — 관련 tool subset만 prompt에
  • Module reuse — 이미 만든 module 캐시 호출 (AgentDistill MCP)

Internalizing Tool Knowledge (2026-05)

Qwen3-4B에서 62% 메모리 절감 · 2.5× 속도 (원문: A100 단일 GPU, ~$62 학습). QLoRA recipe 공개.

한계

tool 정의가 바뀌면 재학습 필요. tool catalog가 자주 바뀌는 환경에서는 운영 비용이 새로운 risk.

Speaker notes

  • 흔히 놓치는 비용 — 모델 자체보다 prompt에 들어간 schema가 더 비싼 경우가 의외로 많다.
  • 쉬운 말: "쓸 수 있는 도구 목록 전체를 매번 함께 보내는 대신, 모델이 외우거나, 골라서 보낸다."
  • 논문 기반 — Internalizing Tool Knowledge (arXiv:2605.17774).
  • 우리 workflow에 외부 tool call은 없지만, "매 호출마다 들어가는 고정 instruction · schema"는 같은 비용 카테고리.

선택지 자체를 줄여 prompt와 오류 확률을 동시에 낮춘다.

Context / trajectory compression 논문 기반

step이 늘수록 누적되는 trajectory를 압축한다 — 모델과 독립인 비용 축.

Trajectory waste 3분류 (AgentDiet)
useless — 현재 task와 무관 redundant — 앞 step에서 이미 처리됨 expired — 한때 필요했으나 현 목표엔 불필요 kept — 현 목표 달성에 필요

AgentDiet (FSE 2026): 입력 토큰 39.9~59.7% ↓ · 전체 연산 비용 21.1~35.9% ↓ (원문: 2개 LLM, 2개 벤치마크, top-performing coding agent)

AgentDiet (FSE 2026)

useless / redundant / expired를 inference-time에 자동 제거 — 학습 불필요.

ACON (arXiv:2510.00615)

observation + interaction history 동시 압축. failure 분석으로 가이드라인 갱신. 소형 compressor distillation. peak token 26–54% ↓, distill 후 정확도 95%+ 보존.

모델을 아무리 작게 만들어도 context가 100K 토큰이면 비용은 남는다. 이 축은 §1·§2의 모델 압축과 독립이다.

Speaker notes

  • 흔한 함정: 모델 크기만 줄이려 하면 context 비용이 그대로 남는다.
  • 쉬운 말: "10개 step을 거친 trajectory에는 이미 필요 없는 정보가 절반쯤 섞여 있다. 그걸 정리한다."
  • 논문 기반 — AgentDiet (FSE 2026), ACON (arXiv).
  • Qwen3 연결: 누적되는 user profile · parent context · candidate note가 길어지는 step이 있다면 직접 관련.

Part 5b · Decoding

Decoding optimization 전에 먼저 확인할 것 — output length 시스템 패턴

어떤 speculative method도 출력 token 수에 비례하는 latency를 완전히 없애진 못한다.

가장 단순한 latency 변수

  • 장황한 reasoning을 출력하면 decode latency가 그만큼 커짐
  • KD 학습 데이터에는 reasoning을 supervision으로 포함하더라도, serving 출력은 짧은 JSON / action으로 제한하는 것이 latency 관점에서 유리

Qwen3 thinking / non-thinking mode

  • Qwen3는 /think · /no_think soft switch 또는 Qwen3-Instruct-2507 / Qwen3-Thinking-2507 variant 선택으로 전환 가능 (Qwen3 공식 문서 / blog)
  • 어려운 case = thinking, 쉬운 case = non-thinking으로 라우팅하는 cascade와 자연 연결

"무조건 non-thinking"은 아니다

reasoning을 줄이면 action quality가 떨어질 수 있다 → schema validity · action accuracy · fallback rate를 함께 측정해야 안전한 trade-off가 가능.

Speaker notes

  • 이 슬라이드는 "decoding optimization을 도입하기 전에" 먼저 점검할 것 — 가장 가벼우면서 효과가 직관적.
  • 쉬운 말: "출력 token 자체를 줄이는 게 어떤 speculative보다 단순한 latency 절감 방법이다."
  • 일반적인 시스템 패턴 — 논문 메서드가 아니라 운영 의사결정.
  • Qwen3 연결: /think · /no_think 전환은 Qwen3 자체 기능이라 추가 학습이 필요 없는 가장 가벼운 카드.

SLIM (arXiv:2403.04260)

큰 teacher가 user behavior sequence CoT rationale 생성 → 작은 student (LLaMA2-7B)가 학습. rationale을 dense vector로 인코딩.

운영 핵심: production output에 rationale 그대로 노출하지 않고 학습 supervision으로만 사용

외삽 금지: 위 사례의 수치(예: PURE의 Amazon dataset 성능)는 평가 metric과 task가 우리와 다르다. "X가 추천에서 좋았다 → 우리에게도 좋다"는 추론을 하지 않는다.

Speaker notes

  • 이 슬라이드는 도메인 참고 사례 — KD 논문도 아니고 결정 근거도 아니다.
  • 왜 보는가: 추천 도메인이 우리와 가까운 문제(누적 사용자 정보, 큰 LM의 출력을 작은 모델에 옮기기)를 먼저 다뤄봤다.
  • 운영상 메시지: rationale은 학습 supervision으로만 쓰고 production output에는 빼는 패턴이 안전 — schema validity / latency를 보호.
  • 외삽 금지 — 한 번 더 강조.

Qwen3-235B teacher 관점 — Qwen3 TR · Every Expert Matters 논문 기반

Qwen3 Technical Report (Alibaba)

  • 소형 모델 학습 = off-policy distillation + on-policy distillation 2단계
  • /think·/no_think soft switch
  • family-internal (tokenizer · chat template 호환)

Every Expert Matters (arXiv:2502.12947)

  • Qwen3-235B-A22B는 MoE → token당 22B만 활성화 → 표준 KD는 22B만 본다
  • KA (Knowledge Augmentation): non-activated expert sampling
  • SAR (Student-Aware Router): student feedback으로 router 학습
  • 적용 조건: teacher 내부 expert · routing 접근 가능해야 함 — API-only면 불가

두 자료의 결합 메시지: Qwen3-235B teacher의 잠재 용량을 student에 전달하려면 (1) family 호환의 운영 이점을 활용하고, (2) MoE 특성을 다루는 KD 메서드를 검토하되, (3) 접근 권한이 그 메서드를 허용해야 한다.

Speaker notes

  • 이 슬라이드는 Qwen3-235B를 teacher로 두는 시나리오의 두 reference.
  • 왼쪽은 공식 recipe, 오른쪽은 MoE-aware KD의 학술 분석.
  • 적용 조건이 핵심: KA / SAR는 teacher 내부 expert sampling이 가능해야 의미가 있다 — hosted endpoint만 쓰는 환경에서는 불가.
  • 한 번 더: 이건 reference이지 채택 결정이 아니다.

MoE에서의 KD — output만 볼 것인가, expert/routing까지 볼 것인가 논문 기반

Qwen3-235B-A22B처럼 MoE teacher를 줄일 때는 teacher 접근 권한에 따라 KD 방식이 달라진다.

논문 / 자료 방향 핵심 아이디어 발표에서 쓸 포인트
Every Expert Matters MoE teacher → dense student non-activated expert도 지식이 있으므로 KA/SAR로 all expert knowledge를 더 잘 활용 Qwen3-235B 같은 MoE teacher를 distill할 때 output만 보면 expert 지식 일부를 놓칠 수 있음
B-Distill MoE teacher → MoE student teacher expertise를 student experts에 균형 있게 전달하고 routing collapse 완화 MoE-to-MoE KD에서는 expert coverage와 router balance가 핵심
Sparse MoE Students dense/MoE teacher → sparse MoE student student 자체를 sparse MoE로 만들어 효율적인 KD 수행 student 구조도 dense만 있는 게 아니라 sparse MoE가 가능
MoEKD for Code Models multi-expert KD expert와 router를 나눠 학습하고 aggregated expert knowledge를 compact model에 distill 단일 teacher source보다 multi-expert aggregation이 robust KD에 유리할 수 있음
Qwen3 Technical Report Qwen family distillation off-policy distillation + on-policy distillation로 lightweight models 학습 Qwen3-235B를 teacher로 둘 때 response KD 후 on-policy/logit KD 흐름을 참고 가능

주의: MoE-aware KD는 router/expert 접근 권한이 있을 때 의미가 크다. API-only 환경에서는 response KD, preference KD, teacher-as-judge 방식이 더 현실적인 black-box KD로 남는다.

EOS_rate와의 관계: MoE-KD 논문이 EOS 취약성을 직접 말하는 것은 아니다. 다만 router/expert 변화가 final logits를 흔들 수 있으므로 EOS_rate, cap_hit_rate, output length distribution은 generation stability 지표로 함께 보는 것이 안전하다.

Speaker notes

  • 이 슬라이드는 MoE teacher를 줄일 때 KD를 세 단계로 나눠 보는 지도다: output-only, logits, router/expert 접근.
  • Every Expert Matters는 "activated expert만 보면 teacher 지식 일부를 놓친다"는 메시지. Qwen3-235B-A22B가 MoE이므로 연결 포인트가 있다.
  • B-Distill이나 Sparse MoE Students는 student를 dense가 아니라 MoE로 설계하는 경우의 참고 사례다. 지금 채택 제안이 아니라 선택지 소개다.
  • EOS_rate는 MoE-KD의 직접 주장은 아니고, final logits와 length behavior가 흔들릴 수 있으니 평가 지표로 같이 보자는 generation stability 관점이다.

Part 7 · 정리

경량화 후 확인해야 할 지표

latency만 보면 함정에 빠진다 — quality risk와 함께 봐야 한다.

Latency 측 (필수)
  • p50 / p95 / p99 latency — 평균이 아니라 tail
  • throughput (tokens / sec, requests / sec)
  • cost per successful task — 단순 latency보다 우선
  • (serving) prefix cache hit rate · KV cache utilization · TTFT
  • (decoding) acceptance rate · tokens per forward
Quality risk 측 (필수)
  • Schema validity rate — JSON / schema 위반 비율
  • Task success rate — end-to-end
  • Multi-turn / multi-step robustness — step 늘릴 때 누적 손실
  • (해당하면) Tool-call accuracy · executable call rate
  • Fallback rate · output repair rate

한 줄 원칙: 일반 벤치마크 (PPL · MMLU)에서 동등해도 schema validity에서 40%p 차이날 수 있다. Compression 결과를 채택하려면 최소 ① schema validity ② task success rate ③ p95 latency ④ cost per successful task 4개를 함께 측정한다.

Speaker notes

  • 지표 분리: latency 측 / quality risk 측 — 두 측을 같이 봐야 진짜 비용 효과가 측정된다.
  • tail (p95 / p99)을 평균으로 대체하면 cascade의 fallback이 가져오는 spike가 안 보인다.
  • 일반적인 시스템 패턴 — 논문 메서드가 아니라 측정 가이드.
  • Qwen3-235B 의존도를 줄이는 어떤 메서드든 이 4개로 측정하지 않으면 채택하지 않는 게 안전한 default.

Takeaways & Profile Agent에 적용 가능해 보이는 지점

결정이 아니라 — 오늘 본 지도가 우리 workflow와 어디서 닿는지의 요약.

5줄 Takeaways
  • "줄인다"는 7개의 다른 비용 — 카드를 고르기 전에 어느 비용인지를 먼저 정한다
  • 일반 KD와 Agent KD는 다르다 — schema · multi-step robustness는 별도 측정
  • serving / runtime · decoding은 모델을 작게 만들지 않는다 — 남은 호출 비용을 친다
  • 가능한 방법은 teacher 접근 권한이 결정한다 — 권한 → 방법 순서
  • latency만 측정하면 함정 — quality risk와 함께 봐야 채택 가치가 보인다
Profile Agent 적용 가능성 (가설)
  • 외부 tool call이 없는 workflow이므로 tool-call accuracy는 직접 risk가 아님. schema validity · merge-decision correctness가 직접 risk
  • 반복되는 instruction · schema · persona 가 있다면 SGLang prefix 재사용 검토 가능성
  • Step 1 / Step 4가 길어지면 output length 제한과 /no_think mode가 가장 가벼운 카드
  • MTP · Medusa · EAGLE은 native 지원 / 학습 권한 확인 전엔 단정 불가
  • 모든 항목은 적용 가능성 — 우선순위 매기지 않는다

이 발표가 다음 단계에 답하지 않는 것: 어떤 방법을 먼저 채택할 것인가, 어떤 step에 어떤 모델을 둘 것인가, KD recipe의 구체 설정. 이는 별도 experiment protocol 단계의 일이다.

Speaker notes

  • 5줄 Takeaways가 오늘 발표의 핵심 — 슬라이드 1장만 기억한다면 이 한 장.
  • Profile Agent 적용 가능성은 짧게: tool-calling agent가 아니므로 tool-call accuracy 외삽 금지. 위험은 schema·merge correctness 중심.
  • "우선순위 매기지 않는다"는 발표 시작 때의 선언과 닫힘 — 오늘의 결론은 "지도"이지 "경로"가 아니다.
  • 다음 단계: experiment protocol 차원에서 가설별로 측정. 그건 별도 세션.
